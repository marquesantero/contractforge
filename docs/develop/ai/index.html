ContractForge AI

This page is kept for older links. The primary documentation now lives in the dedicated ContractForge AI area.

ContractForge AI helps teams turn intent, schemas and project context into reviewable ContractForge artifacts. It is not a replacement for the core planner or adapters. Deterministic validation remains the gate; provider output is advisory unless it passes local schema validation and stays inside the allowed project-spec surface.

Operating model​

The AI package keeps ContractForge boundaries intact:

  • contracts keep source, target, write mode, quality, access and transform semantics;
  • environment YAMLs keep adapter runtime, evidence and deployment settings;
  • adapter packages own Databricks, AWS, Snowflake, Fabric and GCP behavior;
  • model providers cannot move platform-specific settings into ingestion contracts.
  • model providers cannot silently overwrite deterministic identity fields such as connector, source path, target, layer or write mode. Conflicting provider proposals are rejected and recorded in the provider proposal audit.
  • model providers can suggest supported draft contract updates, but fields that can change behavior or deployment, such as transform, shape, quality_rules, annotations, operations and dab_compute, always remain review-required. Unsupported runtime fields, secrets and deployment settings are rejected and audited instead of being written into contracts.

Generate a project​

Use plan-project when the prompt is still rough and you want recommended commands before writing files:

contractforge-ai plan-project \
  --intent "Create an AWS Glue bronze ingestion project from s3://landing/orders into analytics.bronze.b_orders using append." \
  --schema schemas/orders.json \
  --format markdown

Use generate-project when the source, target and adapter target are explicit:

contractforge-ai generate-project \
  --target aws-glue-iceberg \
  --schema schemas/orders.json \
  --project-name orders_aws \
  --connector s3 \
  --source-path s3://landing/orders \
  --target-catalog analytics \
  --target-schema bronze \
  --target-table b_orders \
  --output-dir generated/orders-aws

Use guided-project when one command should plan and scaffold:

contractforge-ai guided-project \
  --intent "Create a bronze to gold medallion project from s3://landing/orders into analytics.bronze.b_orders using hash_diff_upsert." \
  --schema schemas/orders.json \
  --target contractforge-yaml \
  --allow-review-required \
  --output-dir generated/orders

Supported generation targets include contractforge-yaml, contractforge-python, databricks-dab, aws-glue-iceberg, snowflake-sql-warehouse, fabric-lakehouse, gcp-bigquery, dbt and classic-pyspark.

See AI project generation for canonical project.yaml, environment.yaml, connection.yaml and adapter-target details.

Storage and table-format words are not treated as platform choices by themselves. For example, s3://... selects the s3 connector, and iceberg can describe a target table format, but neither one alone forces AWS. If the prompt names AWS Glue, Athena or Lake Formation, AWS becomes the adapter hint. If the prompt names Databricks, Unity Catalog, Asset Bundles or Auto Loader, Databricks becomes the adapter hint. Snowflake, Fabric and GCP/BigQuery service names map to their matching adapter targets. Without an explicit platform, AI planning can recommend multiple adapter paths for the same contract intent. An explicit generation target also acts as a deterministic platform hint: --target aws-glue-iceberg, --target snowflake-sql-warehouse, --target fabric-lakehouse, --target gcp-bigquery and --target databricks-dab add the matching environment scaffold, even when the prompt itself only describes the source and target.

Minimal platform drift​

For portable intent, ContractForge AI can generate one canonical ingestion contract and bind multiple adapters through project.yaml:

environments:
  databricks: environments/databricks.environment.yaml
  aws: environments/aws.environment.yaml
  snowflake: environments/snowflake.environment.yaml
  fabric: environments/fabric.environment.yaml
  gcp: environments/gcp.environment.yaml

execution_order:
  - name: bronze_orders
    contracts:
      databricks: contracts/bronze/orders.ingestion.yaml
      aws: contracts/bronze/orders.ingestion.yaml
      snowflake: contracts/bronze/orders.ingestion.yaml
      fabric: contracts/bronze/orders.ingestion.yaml
      gcp: contracts/bronze/orders.ingestion.yaml

When the selected target is explicitly aws-glue-iceberg, the generator writes an AWS environment scaffold and points AWS project steps at the same canonical contracts. That changes the project and deployment boundary, not the contract semantics. Use adapter-specific contract paths only when platform extensions are required and reviewable.

Validate before deployment​

validate-project-structure checks project shape, environment files, connection YAMLs, split contract bundles and optional adapter planning:

contractforge-ai validate-project-structure generated/orders-aws \
  --adapter aws \
  --format html > project_validation.html

The validator uses ContractForge Core to load bundles and resolve source.type: connection. Ingestion-level source settings override reusable connection YAML defaults after the connection is loaded.

With --adapter, the AI package calls the adapter public planner. SUPPORTED maps to READY; warning-only planner output maps to READY_WITH_WARNINGS with ready=true; blockers, unsupported semantics and unsafe findings remain non-ready.

READY_WITH_WARNINGS is deliberate. It preserves real portability and production-scale warnings without marking a known-good project invalid. For review, --format html renders a rich report with readable cards for findings and adapter-planning evidence instead of wide tables that collapse long paths and warning codes.

See AI validation and parity for project folder validation and Databricks/AWS comparison reports.

Provider routing​

Route work before calling a model provider:

contractforge-ai route-task \
  --intent "Summarize low-risk metadata improvements for this connector" \
  --knowledge-index .contractforge-ai/knowledge.json \
  --prefer-http-only \
  --format markdown

--prefer-http-only can reduce SDK dependency pressure for low-risk advisory tasks. It is not a quality downgrade. Project planning, contract generation and deployable-artifact workflows still require strict structured-output support because their output can affect real ingestion behavior.

Provider quality checks compare model output against deterministic status. If the baseline is NEEDS_DECISIONS, INVALID, UNSAFE, UNSUPPORTED or FAIL, provider summaries and recommendations must not claim the project is ready to deploy, publish, run or proceed.

Review artifact​

Generated projects consolidate review into AI_REVIEW.html. Direct project plans use PROJECT_REVIEW.html, and full project-folder validation can emit project_validation.html through validate-project-structure --format html. These reports contain interpreted intent, generated artifacts, required decisions, context evidence, validation status, adapter evidence and provider guidance where applicable. Treat the HTML report as the approval surface before deployment.

Operational evidence analysis is documented in Evidence intelligence.
