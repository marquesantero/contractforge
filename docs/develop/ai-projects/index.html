AI project generation

This page is kept for older links. The primary documentation now lives in AI project generation.

ContractForge AI generates projects from deterministic intent extraction and core validation. Provider-backed enrichment is optional and cannot override the planner's support status.

Canonical project files​

Generated projects use the current ContractForge structure:

FilePurpose
project.yamlProject inventory, execution order, dependencies and schedule.
environment.yamlRuntime, evidence, artifact destination and guarded adapter parameters.
connections/*.yamlShared connector defaults.
*.ingestion.yamlSource, target, write, transform, quality and dataset-specific source overrides.
*.annotations.yamlTable/column metadata and tags.
*.operations.yamlOwnership, SLA, criticality and runbook metadata.
*.access.yamlGrants, row filters and masks.

Legacy flat fields such as target_table, target_schema, ctrl_schema and top-level source_system are not canonical project output.

Plan before generating​

contractforge-ai plan-project \
  --intent "Create Supabase medallion for Databricks, AWS, Snowflake, Fabric and GCP daily at 6 Sao Paulo time" \
  --schema schemas/products.json \
  --format markdown

The planner extracts connector, source system, layers, write mode, schedule, timezone, governance needs and target platforms. Missing values become explicit required decisions.

Generate AWS​

contractforge-ai generate-project \
  --target aws-glue-iceberg \
  --schema schemas/products.json \
  --project-name supabase_aws \
  --connector postgres \
  --source-path jdbc:postgresql://example/postgres \
  --target-catalog contractforge \
  --target-schema bronze \
  --target-table b_products \
  --mode hash_diff_upsert \
  --output-dir generated/supabase-aws

AWS generation targets the runtime-library model. The generated project should be deployed with contractforge-aws deploy-project; the default is not per-contract generated Glue ingestion code.

Generate Databricks​

contractforge-ai generate-project \
  --target databricks-dab \
  --schema schemas/products.json \
  --project-name supabase_databricks \
  --connector postgres \
  --source-path jdbc:postgresql://example/postgres \
  --target-catalog workspace \
  --target-schema bronze \
  --target-table b_products \
  --mode hash_diff_upsert \
  --output-dir generated/supabase-databricks

Databricks generation uses the same project model and can produce DAB-compatible artifacts when requested.

Generate Snowflake, Fabric or GCP​

Use the matching target while keeping the same source, target, mode, transform and quality intent:

contractforge-ai generate-project \
  --target snowflake-sql-warehouse \
  --schema schemas/products.json \
  --project-name supabase_snowflake \
  --connector postgres \
  --source-path jdbc:postgresql://example/postgres \
  --target-catalog contractforge \
  --target-schema bronze \
  --target-table b_products \
  --mode hash_diff_upsert \
  --output-dir generated/supabase-snowflake

fabric-lakehouse and gcp-bigquery follow the same command shape. Generated adapter projects keep platform-specific runtime values in environments/<adapter>.environment.yaml and contracts under contracts/<adapter>/....

Shared connection YAML​

Use source.type: connection to inherit shared connector defaults:

source:
  type: connection
  connection_path: project://connections/supabase.yaml
  table: public.products
  read:
    fetchsize: 20000

Connection YAML is resolved by the core. Ingestion-level source fields override global connection fields. The adapter receives the resolved source payload.

Deterministic before provider​

Every provider-backed workflow runs deterministic intent extraction, validation and planner checks first. The provider may explain deterministic results or propose review questions, but it must not invent support status or silently change deployable semantics.

Validate generated projects​

After writing a generated project, validate the whole folder before deployment:

contractforge-ai validate-project-structure generated/supabase \
  --adapter databricks \
  --adapter aws \
  --adapter snowflake \
  --adapter fabric \
  --adapter gcp \
  --format html > project_validation.html

The project validator checks project.yaml, environment.yaml, shared connections/*.yaml, split contracts and optional adapter planning. It uses ContractForge Core for bundle loading and connection inheritance; ingestion source values override shared connection defaults after the connection YAML is loaded.

Use the status as a deployment gate:

StatusMeaning
READYNo deterministic blockers or adapter warnings were found.
READY_WITH_WARNINGSThe project is valid and ready=true, but adapter warnings need review.
NEEDS_DECISIONSRequired decisions remain.
INVALID / UNSAFEDo not deploy until fixed.

The HTML report is designed for review. Findings and adapter planning evidence are rendered as cards so long paths, warning codes and planner payloads remain readable.
