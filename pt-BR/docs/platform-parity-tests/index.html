Testes de paridade de plataforma

Esses cenários validam a facilidade com que um contrato ContractForge se move entre Databricks, AWS, Snowflake e Fabric.

A regra é estrita:

  • intenção de ingestão, formato do destino, modo de gravação, qualidade, anotações, operações e a intenção de acesso permanecem as mesmas;
  • apenas alterações de ligação em tempo de execução: local de origem, contrato de ambiente e extensões de propriedade do adaptador;
  • se uma plataforma não puder preservar a mesma semântica, o adaptador deverá retornar SUPPORTED_WITH_WARNINGS, REVIEW_REQUIRED ou UNSUPPORTED.

Validação local​

Execute os testes do contrato de paridade:

uv run pytest tests/test_platform_parity_contracts.py

Generate a machine-readable report:

uv run python -m tools.platform_parity.report

O relatório inclui todos os status do adaptador, nomes de artefatos e a localização exata delta de plataforma permitido pelo teste.

Gere dados determinísticos JSONL para corridas de smoke reais:

uv run python -m tools.platform_parity.data --output .tmp/platform-parity-data

Faça upload de cada diretório de cenário gerado para o caminho de origem mostrado por tools.platform_parity.report:

  • Databricks: dbfs:/tmp/contractforge/parity/<dataset>/;
  • AWS: s3://contractforge-parity-us-east-1/data/<dataset>/;
  • Snowflake: @"CONTRACTFORGE_TEST_DB"."PUBLIC"."CF_PARITY_DATA"/<dataset>/.
  • Fabric: Files/contractforge/parity/<dataset>/.

Validação de execução real​

Os mesmos contratos de cenário devem ser usados ​​para Databricks reais, AWS, Snowflake e, quando validado, Fabric smoke corre. Não edite manualmente o semântica de ingestão por plataforma.

Diferenças de tempo de execução permitidas:

  • Caminho de origem Databricks, por exemplo dbfs:/tmp/contractforge/parity/...;
  • Caminho de origem AWS, por exemplo s3://contractforge-parity-us-east-1/data/...;
  • Caminho de origem do arquivo preparado Snowflake, por exemplo @"CONTRACTFORGE_TEST_DB"."PUBLIC"."CF_PARITY_DATA"/...;
  • Fabric OneLake Caminho dos arquivos, por exemplo Files/contractforge/parity/...;
  • catálogo/esquema de evidências do ambiente Databricks;
  • banco de dados de evidências do ambiente AWS e padrões de implantação de trabalho Glue;
  • Ligações de armazém/esquema/estágio do ambiente Snowflake;
  • Fabric environment tenant/workspace/lakehouse/warehouse bindings;
  • AWS extensions.aws.iceberg.warehouse;
  • Snowflake extensions.snowflake.explain_enabled.

Critérios de aceitação para uma corrida real:

  • tabelas de destino são criadas em todas as plataformas;
  • tabelas principais de evidências/controle são criadas por meio dos adaptadores;
  • ctrl_ingestion_runs registra o mesmo status lógico por cenário;
  • rows_read, rows_written, rows_quarantined e correspondência de status de qualidade dados do cenário esperado;
  • A governança somente para revisão do AWS permanece somente para revisão até Lake Formation a equivalência é validada para um mecanismo consumidor concreto.
  • Snowflake JSON aninhado com formato pesado permanece necessário para revisão até SQL/Snowpark a paridade de forma é validada em relação ao comportamento Spark/Glue.

USGS GeoJSON Adaptador cruzado E2E​

A página canônica de paridade ponta a ponta entre adaptadores é Teste contratos entre adaptadores. Essa página permanece focada no Databricks/AWS/Snowflake estável lado a lado. O exemplo mais amplo do USGS e o relatório de paridade compartilhada adicionam o atual Fabric subconjunto notebook-first sem alterar a intenção principal do contrato.

Este cenário abrange:

  • intenção de origem pública USGS GeoJSON;
  • raw GeoJSON bronze landing;
  • Análise features[], nivelamento de array, extração de coordenadas e evento tipado fields;
  • deterministic event deduplication by earthquake_id;
  • regras de qualidade prateada para IDs de eventos, carimbos de data/hora, latitude e longitude;
  • agregados diários de ouro e por faixa de magnitude;
  • ligação de destino nativo do adaptador para tabelas nativas Delta, Iceberg e Snowflake;
  • evidências da tabela de controle para execuções, qualidade, quarentena e erros.

Contagens de linhas esperadas para a validação de feed USGS GeoJSON de 2,5 dias:

OutputLinhas
bronze GeoJSON raw response1
silver earthquake events30
gold daily summary2
gold magnitude bands3

Os adaptadores mantêm a mesma intenção de ingestão e alteram apenas a ligação nativa:

TópicoDatabricksAWSSnowflake
Fonte de bronzeSolicitação REST APISolicitação REST APISolicitação REST API com integração de acesso externo Snowflake
Tempo de execuçãoTempo de execução do adaptador DatabricksRunner estável AWS GlueRunner de procedimento SQL Snowflake
Formato de tabelaDeltaIcebergTabela nativa Snowflake
EvidênciaTabelas de controle DeltaTabelas de controle Iceberg consultadas pelo AthenaTabelas de controle Snowflake

A reivindicação de paridade é intencionalmente restrita: GeoJSON valida o mesmo medalhão família de contrato em todos os três adaptadores. A paridade JDBC/database-source permanece um preocupação específica do conector separada e não é a página de prova de adaptador cruzado. A execução do procedimento hospedado Snowflake REST requer que CF_USGS_REST_ACCESS permita HTTPS de saída para earthquake.usgs.gov:443.
