Critérios GA do adaptador Databricks

Propósito​

Este documento define as condições verificáveis ​​sob as quais contractforge-databricks é promovido de Development Status:: 3 - Alpha to General Availability (Development Status :: 5 - Production/Stable, versão do pacote 1.0.0).

É um portão, não uma descrição de recursos. O escopo implementado é documentado em databricks-adapter.md. O funcional parity matrix lives in databricks-contractforge-parity.md. Este arquivo os complementa listando as condições devem ser verdadeiras e as como verificar cada um.

Um critério é considerado atendido somente quando a etapa de verificação é implementada, rodando em CI e verde por pelo menos quatro ciclos semanais consecutivos após o criterion is added.

Escopo deste portão​

Esta porta cobre apenas o pacote contractforge-databricks. AWS, Snowflake e os pacotes de IA têm seus próprios portões. Núcleo (contractforge-core) GA está vinculado para Databricks GA porque o adaptador Databricks é a referência implementação: quando Databricks atinge 1.0.0, o núcleo é promovido para 1.0.0 na mesma janela de lançamento.

Este portão não requer:

  • adaptadores diferentes de Databricks para alcançar GA;
  • Lakeflow AUTO CDC deve ser validado em tempo de execução para o estado atual (apenas o histórico está no escopo);
  • mutação do espaço de trabalho através do contractforge-databricks CLI (implantação remains a customer-owned concern);
  • a GUI;
  • streaming contínuo. Somente o comportamento disponível agora/fluxo limitado está em escopo, consistente com as especificações do adaptador.

O seguinte deve estar verde antes que qualquer critério do GA seja avaliado. Estes são não específico do GA; eles são invariantes de repositório.

Se alguma pré-condição estiver vermelha, a avaliação do GA será pausada. Corrija a pré-condição primeiro; não renuncie a isso.

GA Criteria​

Cada critério abaixo tem três partes: o que deve ser válido, como verificar e status. O status é not_started, in_progress, met ou waived (com razão). As isenções são caso a caso, nunca gerais.

1. Modos de gravação – Execução ponta a ponta​

O que deve ser mantido. Todos os seis modos de gravação portáteis são executados de ponta a ponta um espaço de trabalho Databricks real com o mecanismo suportado selecionado por write_modes/strategy.py:choose_write_strategy:

  • append → Acrescento Delta (nativo)
  • overwrite → substituição de Delta (nativo)
  • upsert → Databricks SQL MERGE (nativo)
  • hash_diff_upsert → ContractForge hash-diff Delta MERGE (algorithm)
  • historical → ContractForge historical Delta MERGE (algorithm); Lakeflow AUTO CDC permanece renderizado pelo adaptador, mas não está na porta de execução GA
  • snapshot_reconcile_soft_delete → ContractForge snapshot soft-delete Delta MERGE (algorithm)

Each successful run produces:

  • uma tabela de destino Delta no nome UC de três partes do contrato;
  • uma linha em ctrl_ingestion_runs com status = 'SUCCEEDED';
  • métricas de gravação normalizadas (rows_inserted / rows_updated / rows_deleted / rows_expired / rows_affected) as defined in databricks-adapter.md §Gravar Métricas;
  • stage durations recorded in stage_durations_json.

Como verificar. O trabalho de smoke do espaço de trabalho ativo (consulte a seção 11) executa um contrato de fixação por modo de gravação. Testes off-line existentes (tests/test_databricks_delta_basic_execution.py, tests/test_databricks_sql_merge_execution.py, tests/test_databricks_hash_diff_execution.py, tests/test_databricks_scd2_execution.py, tests/test_databricks_snapshot_execution.py, tests/test_databricks_replace_partitions_execution.py) stay green in PR CI.

Fora do escopo do GA. Os modos de gravação custom:* são fornecidos como uma extensão estável apontam, mas não estão no portão GA; os consumidores carregam sua própria cobertura.

Status. not_started.

2. Conectores de Origem – Status Declarado Honrado​

O que deve ser válido. Todas as fontes declaradas por sources/support.py:list_databricks_source_support behaves according to its declared status:

Status declaredGA requirement
SUPPORTEDUm equipamento de espaço de trabalho ativo por família de origem é executado de ponta a ponta (leitura + gravação em Delta)
REVIEW_REQUIREDO renderizador emite o artefato de revisão esperado e um aviso de planejamento; nenhuma execução silenciosa
UNSUPPORTEDAdaptador retorna um bloqueador de planejamento; nenhum artefato é emitido

Famílias de origem cobertas pelo GA:

  • incremental_files (Auto Loader)
  • jdbc family (postgres, mysql/mariadb, sqlserver, oracle, redshift)
  • catalog (table, view, sql, delta_table, iceberg_table)
  • http_file / http_csv / http_json / http_text
  • rest_api (bounded resolver only)
  • bounded streams (kafka_bounded, eventhubs_bounded)
  • delta_share
  • batch file/object-storage (csv, json/jsonl/ndjson, parquet, delta, orc, text, avro, xml, s3, adls, azure_blob, gcs)
  • native_passthrough (somente artefato REVIEW_REQUIRED)

Driver responsibilities (e.g. Oracle JDBC driver, XML format jar) remain a preocupação de implantação e estão explicitamente fora do escopo do portão do adaptador, mas o espaço de trabalho utilizado pela defumação deve fornecê-los para que as luminárias possam executar.

Como verificar. A smoke viva opera uma luminária por família. Testes off-line (tests/test_databricks_*_source.py, tests/test_databricks_source_*.py, tests/test_databricks_runtime_sources.py) stay green.

Status. not_started.

3. Política de Esquema – Planejar, Aplicar, Auditar​

O que deve ser válido. As três políticas de esquema principais se comportam conforme especificado:

  • additive_only — aplicam-se alterações aditivas; quebras estruturais falham com a blocker; ctrl_ingestion_schema_changes records each applied change
  • permissive — todas as diferenças detectadas são registradas, mas não bloqueadas
  • fail — qualquer diferença detectada falha na execução antes da gravação

Os registros de alteração de esquema incluem change_type, column_name, previous_type, current_type e o ID de execução.

Como verificar. tests/test_databricks_schema_diff.py, tests/test_databricks_schema_policy.py e um fumo vivo por apólice comportamento.

Status. not_started.

4. Regras de Qualidade – Renderização, Execução, Quarentena, Evidência​

O que deve ser válido. Todos os sete tipos principais de regras de qualidade — required_columns, not_null, unique_key, accepted_values, row_count_minimum, max_null_ratio, expression — executado em Databricks e o resultado is recorded:

  • ctrl_ingestion_quality recebe uma linha por regra por execução com status, failed_records e metadados de regras;
  • on_quality_fail = fail aborta a gravação;
  • on_quality_fail = warn registra a falha e continua;
  • on_quality_fail = quarantine grava linhas com falha no contrato destino de quarentena antes que a gravação principal prossiga.

Como verificar. tests/test_databricks_quality_*.py e uma smoke viva que exercita cada ramificação on_quality_fail.

Status. not_started.

5. Governança e anotações – Aplicar e auditar​

O que deve ser válido. Unity Catalog SQL é renderizado e aplicado para:

  • comentários de tabela, comentários de coluna, tags de tabela, tags de coluna, aliases, PII metadados e marcadores de descontinuação (anotações);
  • concessões, filtros de linha e máscaras de coluna (acesso);
  • owner review notes.

Comportamentos necessários para GA:

  • anotações policy=ignore|warn|falhar produzir o resultado documentado e linha de auditoria em ctrl_ingestion_annotations;
  • acesse mode = ignorar|validate_only|apply produz o resultado documentado e linha de auditoria em ctrl_ingestion_access;
  • acessar on_drift=fail rejeita desvio detectado quando o runner suporta query();
  • acessar revoke_unmanaged=true requer o tempo de execução documentado confirmation before issuing REVOKE.

Como verificar. tests/test_databricks_annotations_sql.py, tests/test_databricks_governance_application.py, tests/test_databricks_governance_log.py, tests/test_databricks_access_audit_sql.py e uma smoke viva que vira cada modo.

Status. not_started.

6. Metadados de Operações – Registrados​

O que deve ser mantido. Metadados de operações (criticalidade, frequência esperada, atualização SLA, sinalizadores de alerta, runbook URL, propriedade, proprietários, grupos, tags) é gravado em ctrl_ingestion_operations quando declarado, e retorna NOT_CONFIGURED quando ausente.

Como verificar. tests/test_databricks_operations_application.py, tests/test_databricks_operations_sql.py e uma luminária de smoke com total metadados de operações.

Status. not_started.

7. Armazenamentos de Evidências – Esquemas, DDL, Migração​

O que deve ser válido. Todas as doze tabelas de evidências/controle podem ser criadas, escrito e migrado adicionalmente em um espaço de trabalho real:

ctrl_ingestion_runs, ctrl_ingestion_errors, ctrl_ingestion_quality, ctrl_ingestion_state, ctrl_ingestion_locks, ctrl_ingestion_schema_changes, ctrl_ingestion_lineage, ctrl_ingestion_streams, ctrl_ingestion_annotations, ctrl_ingestion_access, ctrl_ingestion_operations, ctrl_ingestion_explain.

A migração SQL renderizada por state/migrations.py é idempotente em um workspace que já possui a versão anterior da tabela de controle instalada.

Como verificar. tests/test_databricks_evidence_ddl.py, tests/test_databricks_evidence_sql.py, tests/test_databricks_control_table_schema_parity.py, tests/test_databricks_state_sql.py e uma smoke viva que percorre o migração SQL em um espaço de trabalho com tabelas anteriores instaladas.

Status. not_started.

8. Lineage — OpenLineage Event Per Run​

O que deve ser mantido. Para cada execução bem-sucedida, o tempo de execução emite um OpenLineage-compatible record persisted through ctrl_ingestion_lineage com namespace databricks://<catalog>, resultado de execução correspondente eventType (COMPLETE ou FAIL), facetas do esquema de entrada/saída e a linha de saída contam como uma faceta métrica de qualidade de dados.

Como verificar. tests/test_databricks_lineage.py e smoke viva emite pelo menos um registro válido por dispositivo de modo de gravação.

Status. not_started.

9. Cost Signals — Reportable​

O que deve ser mantido. cost/sql.py:render_operational_cost_query produz SQL que é executado nas tabelas de evidências preenchidas e retorna um conjunto de resultados não vazio quando chamado com dbu_per_hour válido, currency_per_dbu e currency.

Esta é uma estimativa de custos operacionais, não uma reconciliação de faturamento do fornecedor; isso permanece fora do escopo explícito.

Como verificar. tests/test_databricks_cost.py e smoke ao vivo a consulta de custo após pelo menos uma execução bem-sucedida.

Status. not_started.

10. Parity Scenarios — Validated Live​

O que deve ser válido. Todos os oito cenários declarados em adapters/databricks/src/contractforge_databricks/parity/scenarios.py ter uma execução de espaço de trabalho ao vivo correspondente que produza o resultado esperado:

  • seis cenários must_match (3× SQL MERGE do estado atual, 3× ​​Lakeflow histórico AUTO CDC) produzem métricas que correspondem à execução de referência ContractForge Delta;
  • one unsupported scenario (hash_diff_auto_cdc_non_equivalence) is correctly blocked at tempo de planejamento;
  • one intentional_difference scenario (snapshot_reconcile_soft_delete_auto_cdc_difference) é renderizado com o bloqueador declarado registrado na evidência de corrida.

Como verificar. O trabalho de smoke ao vivo executa o harness de paridade semanalmente. Off-line test tests/test_databricks_parity_catalog.py continua a impor o catalog shape in PR CI.

Status. not_started.

Live Workspace Smoke​

O portão GA depende de um trabalho de smoke recorrente. Decisões para o primeiro release window:

ItemDecision
WorkspaceO mesmo espaço de trabalho já utilizado para validar examples/real-world/supabase-jdbc-medallion ponta a ponta em Databricks e AWS.
Catálogo e esquemaReutiliza o esquema de evidência de exemplo existente (cf_supabase_jdbc_e2e_v2_ops) e o catálogo de destino (workspace). Nenhuma nova nomenclatura.
FrequencyWeekly cron (Sunday 06:00 UTC) plus on-demand workflow_dispatch. Daily is intentionally out of scope due to compute cost.
Acionar fluxo de trabalhoNovo .github/workflows/databricks-smoke.yml (a ser criado como parte da Fase 1; ainda não presente).
SecretsDATABRICKS_HOST, DATABRICKS_TOKEN, DATABRICKS_WAREHOUSE_ID injetados como segredos do repositório; não ecoou; nunca logado.
Failure handlingUma execução semanal com falha bloqueia o próximo ciclo de revisão do GA até que a execução fique verde ou a falha seja explicitamente dispensada com um motivo registrado.
TeardownO trabalho smoke descarta apenas objetos criados no esquema dedicado. Os artefatos de exemplo existentes são preservados.
Cost envelopeDocumentado no fluxo de trabalho de smoke. Uma execução que exceda o envelope é investigada antes do próximo ciclo.

O fluxo de trabalho de smoke executa uma matriz de teste fixa derivada dos critérios do GA acima. Novos critérios exigem um teste off-line e uma entrada na matriz de smoke antes que o critério possa ser marcado como met.

Política de alterações significativas pós-GA​

Once contractforge-databricks reaches 1.0.0:

  • Public API surface (everything documented in databricks-adapter.md e acessível sem um underscore prefix) follows SemVer.
  • Breaking changes require a 2.0.0 bump; no exceptions absent a documented security carveout.
  • As descontinuações atingem um menor antes da remoção, com um DeprecationWarning emitido na importação ou no momento da chamada. Janela mínima de depreciação: 90 dias.
  • Os esquemas de tabelas de evidências/controle evoluem de forma aditiva. Migrações destrutivas não são permitidos em versões secundárias.
  • As superfícies de extensão sob extensions.databricks.* seguem o política de parâmetros do adaptador e são regidos pelo mesmo contrato SemVer.
  • As declarações de capacidade em adapters/databricks/src/contractforge_databricks/capabilities/ form parte do API público. Renomear um recurso é uma mudança significativa.
  • Parity scenarios in adapters/databricks/src/contractforge_databricks/parity/scenarios.py pode ser adicionado sem colisão. Alterando um valor expectation ou removing a scenario is a breaking change.

A política é aplicada pelo tests/test_publication_packaging.py existente verificações e por revisão durante a liberação.

De-GA Criteria​

O status 1.0.0 é revogado e o pacote retorna à semântica 0.x, se alguma das seguintes situações se tornar verdadeira:

  • quatro corridas de smoke semanais consecutivas falham sem uma renúncia aceita;
  • a security vulnerability requires a breaking change inside a minor;
  • um modo de gravação que estava na porta GA é removido ou sua semântica alterações contratuais;
  • o isolamento da plataforma principal foi violado (uma versão contractforge-core começa a importar uma plataforma SDK).

Um evento de-GA deve ser registrado no pacote CHANGELOG com o triggering condition cited.

1.0.0 Release Checklist​

Lista de verificação binária para a tag de lançamento real v1.0.0-databricks:

  • [] Todas as seções 1–10 acima marcadas como met com histórico de smoke verde para at least four consecutive weeks.
  • All inherited preconditions green.
  • pyproject.toml updated: version = "1.0.0" e Development Status :: 5 - Production/Stable.
  • adapters/databricks/CHANGELOG.md includes a [1.0.0] entry que lista o limite de estabilidade API e faz referência a esta porta document.
  • [] databricks-adapter.md revisado e qualquer desvio da implementação reconciliado.
  • databricks-contractforge-parity.md reviewed.
  • docs/specs/api-stability.md updated to reflect Databricks as a superfície estável.
  • [] Tabela de status README.md para o adaptador Databricks alterada de "Reference implementation" to "GA".
  • [] contractforge-core passou para 1.0.0 na mesma janela de lançamento; 4 - Beta becomes 5 - Production/Stable.
  • [] Lançamento do PyPI enviado via release.yml com tag v1.0.0-databricks (e v1.0.0-core); instalar smoke, correr Python 3.10/3.11/3.12 Linux + Windows; ambas as páginas são renderizadas corretamente.
  • [] Trusted Publisher confirmado ativo para ambos os pacotes.
  • [] Lançamento anunciado na página do projeto e na documentação índice. As páginas dos adaptadores AI, AWS e Snowflake afirmam claramente que são ainda não está no portão GA.

Não objetivos deste portão​

  • GA de contractforge-aws, contractforge-snowflake ou contractforge-ai (each has its own gate).
  • Uma garantia de que todo contrato que passa no AWS ou Snowflake o adaptador também funciona em Databricks; a portabilidade permanece no nível do contrato, não equivalência de adaptador.
  • Validação de tempo de execução ao vivo do estado atual do Lakeflow AUTO CDC, personalizado semelhante ao EMR motores, streaming contínuo ou adaptadores Fabric / GCP.
  • Um planejador ou tempo de execução de orquestração de propriedade de ContractForge.

Itens abertos antes que o portão esteja ativo​

  • Fluxo de trabalho .github/workflows/databricks-smoke.yml a ser criado.
  • Envelope de custo por ciclo de smoke a ser calibrado em relação ao espaço de trabalho.
  • Primeira entrada de isenção real para validar o formato do registro em docs/specs/databricks-ga-waivers.md se uma renúncia for necessária.
