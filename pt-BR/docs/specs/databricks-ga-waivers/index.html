Databricks GA Criteria — Waiver Registry

Propósito​

A porta Databricks GA documentada em databricks-ga-criteria.md is intentionally estrito. Uma renúncia é a exceção documentada que permite ao 1.0.0 janela de liberação para prosseguir quando um critério não puder ser atendido pelo corte date.

A waiver is never blanket. Each one is scoped to a single criterion, has a proprietário responsável nomeado, uma data de expiração e um plano de remediação. O A revisão do GA usa este arquivo como fonte da verdade para a qual as exceções são currently in force.

Se este arquivo não listar uma isenção para um critério específico, isso critério é tratado como not_met para fins de liberação checklist.

Quando uma renúncia é permitida​

Uma renúncia pode ser concedida somente quando todos os seguintes itens forem válidos:

  • o critério não é um controle de segurança ou correção de dados;
  • a lacuna é documentada com evidências reproduzíveis (identificações de teste, smoke links ou logs do espaço de trabalho);
  • existe um plano de remediação concreto com uma data prevista não superior a 90 dias após o registro da renúncia;
  • não há solução alternativa que introduza degradação semântica silenciosa;
  • o comportamento afetado é documentado como uma limitação conhecida em o CHANGELOG voltado para o usuário ou atrás de um sinalizador de recurso.

Isenções são proibidas para:

  • isolamento da plataforma principal (qualquer violação bloqueia o GA completamente);
  • limites de independência do adaptador;
  • persistência de evidências para ctrl_ingestion_runs e ctrl_ingestion_errors;
  • modos de gravação produzindo contagens incorretas de linhas ou perda de linhas;
  • aplicativo de governança que concede silenciosamente mais acesso do que o declarado.

Waiver Lifecycle​

Each waiver moves through three states:

  • active — recorded, in force, blocking nothing.
  • expired — data de expiração vencida e não renovada. O lançamento checklist deve tratar o critério subjacente como not_met até que o a renúncia for renovada ou o critério for atendido.
  • revoked — retirado antes do vencimento, seja porque o critério foi cumprida ou porque a condição subjacente à renúncia já não holds.

Depois que uma renúncia é registrada, sua linha no registro abaixo nunca é excluído. É marcado apenas como expired ou revoked com data e motivo. Isso preserva o histórico de auditoria.

Approval Requirements​

  • O mantenedor do projeto aprova qualquer renúncia e é o proprietário padrão unless another owner is explicitly assigned.
  • Um segundo revisor é necessário para isenções que afetam evidências, critérios de governação ou de qualidade (secções 4, 5, 7 dos critérios da AG).
  • As aprovações são registradas na linha do registro, não nos comentários de PR.
  • As renovações zeram o relógio de 90 dias e são registradas como uma nova linha, não por editando o original.

Waiver Entry Format​

Cada renúncia é uma linha na tabela de registro abaixo, mais uma subseção em "Isenções Ativas" com todos os detalhes.

A subseção de detalhes deve conter:

  • Critério: o título exato da seção de databricks-ga-criteria.md.
  • Estado: active, expired ou revoked.
  • Recorded: ISO date.
  • Expira: data ISO (90 dias no máximo a partir do registro).
  • Proprietário: identificador GitHub do responsável.
  • Aprovador(es): identificador(es) GitHub do(s) mantenedor(es) de aprovação.
  • Motivo: por que o critério não pode ser atendido pelo ponto de corte do GA.
  • Evidências: links para IDs de teste, corridas de smoke ou registros.
  • Solução alternativa: qual comportamento os consumidores observam nesse meio tempo; não deve introduce silent semantic degradation.
  • Plano de remediação: etapas concretas e data prevista para fechamento.
  • Política de renovação: se a renovação é antecipada e sob que condition.
