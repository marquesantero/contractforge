Paridade da tabela de controle

Propósito​

As tabelas de controle são um contrato de evidência fundamental doContractForge.

O nome principal ainda é modelo de evidência, mas os adaptadores devem preservar o vocabulário da tabela de controleContractForgeonde quer que a plataforma possa suportá-lo. Os adaptadores de plataforma podem persistir os dados como tabelasDelta, tabelasIceberg, tabelasBigQuery, tabelas de auditoriaSnowflake, tabelasFabricLakehouseou artefatos de armazenamento de objetos, mas os campos semânticos abaixo são a superfície de auditoria canônica.

Esta especificação de paridade define:

  • evidências canônicas/tabelas de controle
  • grupos de colunas canônicas da implementação atual doContractForge
  • equivalência de armazenamento de plataforma
  • status de portabilidade para campos que são portáteis, específicos do adaptador, com revisão obrigatória ou sem suporte

Para o mapeamento denso coluna por coluna para a fonte de evidência nativa de cada plataforma (Databricks,AWS,GCP,Snowflake,Fabric), consulte a Matriz de mapeamento de evidências. Para obter um mapeamento específico doSnowflake, apoiado por documento oficial, dos parâmetros principais e campos da tabela de controle, consulte capacidade doSnowflakee paridade de evidências.

Status Legend​

StatusMeaning
PORTABLEO campo tem o mesmo significadoContractForgeem todas as plataformas.
ADAPTER_MAPPEDO campo tem significado portátil, mas o adaptador deve traduzir valores nativos da plataforma.
REVIEW_REQUIREDO campo pode ser capturado, mas a equivalência depende do design da plataforma ou da integração do tempo de execução.
PLATFORM_SPECIFICO campo é uma evidência útil, mas só tem significado nativo em algumas plataformas.
OPTIONALO campo deve ser capturado quando disponível; os planos não devem depender disso.

Requisitos do adaptador​

  1. Os adaptadores de produção devem declarar uma estratégia de persistência de evidências.
  2. ctrl_ingestion_runsou um livro-razão de execução consultável equivalente é necessário para planos de produção.
  3. As gravações de evidências nunca devem abandonar silenciosamente a semântica doContractForge.
  4. Valores sensíveis de origem/autenticação/tempo de execução devem ser editados antes da persistência.
  5. As métricas específicas da plataforma devem ser normalizadas em contadores de linhaContractForgesempre que possível.
  6. As colunas de versão/tempo de execução usam nomes neutros em termos de plataforma:table_version_before/table_version_after(STRING),last_table_version(STRING),runtime_entrypoint,engine_version. Cada adaptador mapeia seu valor nativo (versãoDelta, ID de instantâneoIceberg, ID de trabalhoBigQuery, etc.) neles e mantém o valor nativo bruto emoperation_metrics_json.
  7. Cada adaptador deve documentar quais colunas canônicas são nativas, mapeadas, nulas ou necessárias para revisão.
