Capacidade Snowflake e paridade de evidências

Propósito​

Esta especificação define como um futuro adaptador contractforge-snowflake deve mapear ContractForge Parâmetros principais do contrato e campos de evidência/tabela de controle para Recursos nativos do Snowflake.

O objetivo não é tornar o núcleo consciente do Snowflake. O objetivo é dar ao Adaptador Snowflake um alvo de implementação pesquisado:

Semântica do contrato ContractForge
  -> declaração de capacidade Snowflake
  -> implementação Snowflake SQL / task / stream / policy
  -> tabelas de evidência canônicas ContractForge preenchidas com valores Snowflake

As fontes deste documento são a documentação oficial do Snowflake. O adaptador deve verificar o comportamento em uma conta Snowflake real antes de alterar qualquer item REVIEW_REQUIRED para SUPPORTED.

Sub-alvos do adaptador​

O adaptador Snowflake não deve declarar um recurso vago de snowflake. Isto deve declarar subalvos explícitos.

SubalvoStatus alvoPropósito
snowflake_sql_warehousePrimárioExecução de SQL na computação do warehouse. Lida com gravações de tabela, SQL, estágio de arquivo, qualidade, mesclagem e evidências.
snowflake_task_graphCaminho de implantação principalRenderiza artefatos de implantação de tarefas/gráficos de tarefas para execução agendada de projetos.
snowflake_snowpipeRevisão primeiroLida com o carregamento contínuo de arquivos preparados onde a semântica do Snowpipe preserva o contrato.
snowflake_streams_tasksRevisão primeiroLida com a tabela CDC e fluxos de trabalho incrementais tabela a tabela.
snowflake_snowparkFuturoLida com transformações complexas ou comportamento do conector que não é limpo apenas no SQL.
snowflake_native_passthroughFuturoDelega para conectores nativos Snowflake ou ingestão de aplicativos nativos/de mercado.

Estratégia de armazenamento de evidências​

O adaptador Snowflake deve persistir evidências nas tabelas Snowflake com o mesmo nomes canônicos e colunas definidos por contractforge_core.evidence. Estes são Tabelas de evidências ContractForge, não visualizações de uso da conta Snowflake. Snowflake visualizações nativas fornecem valores; eles não substituem a evidência ContractForge modelo.

Recommended default:

environment:
  adapter: snowflake
  evidence:
    database: CONTRACTFORGE
    schema: CF_EVIDENCE

As tabelas de controle devem ser somente anexadas, exceto superfícies auxiliares de bloqueio/estado onde o O adaptador possui uma estratégia de simultaneidade claramente documentada.

ACCESS_HISTORY é uma fonte de auditoria; o adaptador ainda deve emitir um Evento de linhagem ContractForge durante a execução para que os painéis não dependam apenas de Account Usage latency.

A execução em tempo de execução agora grava um estilo ContractForge/OpenLineage imediato Evento COMPLETE com ID de execução do contrato, referência de origem, tabela de destino, linha contagem e IDs de consulta Snowflake. A reconciliação nativa ACCESS_HISTORY continua sendo um caminho de uso da conta atrasado: ele investiga por QUERY_TAG run_id estruturado e retorna PENDING sem escrever linhas até que as linhas de linhagem nativa correspondentes sejam published.

ctrl_ingestion_cost​

ContractForge fieldFonte Snowflake

O custo é um sinal operacional, não uma reconciliação de faturamento. Uso da conta latência significa que a evidência de custo pode ser anexada por um comando de reconciliação posterior. Se o Uso da conta ainda não tiver produzido linhas correspondentes, a reconciliação reportará um status pendente sem inserir evidências duplicadas. O mesmo caminho pendente é usado quando a função de execução não consegue ler o uso da conta, com um aviso para operadores podem conceder a função de banco de dados Snowflake necessária separadamente do tempo de execução ingestion privileges.

ctrl_ingestion_metadata​

O adaptador preenche os metadados do componente ContractForge, versão principal, adaptador versão, versão do esquema de controle, conta/região Snowflake quando configurada, padrões de warehouse/função e sinalizadores de recursos de tempo de execução. Valores sensíveis devem ser redacted.

Query Tag Requirement​

O adaptador Snowflake deve definir um QUERY_TAG estruturado para cada instrução que ele possui. No mínimo:

{
  "product": "contractforge",
  "adapter": "snowflake",
  "run_id": "<contractforge-run-id>",
  "project": "<project-name>",
  "target": "<database.schema.table>"
}

Isso fornece QUERY_HISTORY, QUERY_ATTRIBUTION_HISTORY, ACCESS_HISTORY e tarefa de diagnóstico de uma chave de junção estável de volta à evidência ContractForge.

Revise os limites​

O planejador Snowflake inicial deve retornar REVIEW_REQUIRED ou SUPPORTED_WITH_WARNINGS para estas áreas até ser comprovado por testes reais:

  • incremental_files usando Snowpipe ou histórico de cópia.
  • Kafka/Event Hubs bounded replay.
  • histórico com linhas que chegam atrasadas, exclusões e janelas de validade sobrepostas.
  • Exclusão reversível de instantâneo sem um instantâneo de origem completo comprovado.
  • Políticas de acesso e mascaramento de linha com lógica específica do principal.
  • Geração de políticas de mascaramento baseadas em tags.
  • Operações shape complexas em cargas VARIANT aninhadas.
  • Quarentena para regras de qualidade agregadas.
  • Ampliação de tipo e alterações de esquema incompatíveis.
  • Atribuição de custos quando a latência de uso da conta atrasa linhas de custo de consulta.

Testes de aceitação para um futuro adaptador​

  1. Planeje o projeto canônico do medalhão Supabase contra Snowflake e produza exact warnings/review items.
  2. Renderize SQL para fontes de tabela, SQL e arquivos preparados sem importar um Cliente Snowflake no núcleo.
  3. Execute acréscimo, substituição e atualização do estado atual em uma conta Snowflake real.
  4. Verify ctrl_ingestion_runs, state, quality, quarantine, errors, schema_changes, annotations, access, lineage, operations, explain e as tabelas cost usam o esquema principal.
  5. Join ContractForge run_id to Snowflake QUERY_HISTORY through QUERY_TAG.
  6. Capture evidências de tarefas para um projeto agendado com pelo menos uma dependência.
  7. Aplique comentários/tags e valide-os através do TAG_REFERENCES.
  8. Aplique políticas de acesso e mascaramento de linha e valide-as por meio de POLICY_REFERENCES.
  9. Prove que o histórico permanece REVIEW_REQUIRED até que o projeto declare o políticas de atraso, exclusão e data de vigência necessárias para uma execução segura.
