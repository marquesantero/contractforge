Lista de verificação de revisão técnica do adaptador

Status: lista de verificação de execução aprovada

Objetivo: definir a lista de verificação de revisão que ContractForge usará antes de adicionar ou promoção de adaptadores de plataforma. Cada adaptador é revisado como um independente módulo. Não assuma estado compartilhado, credenciais compartilhadas, clientes de tempo de execução compartilhados, ou comportamento de implantação compartilhada entre adaptadores.

Initial scope:

  1. Adaptador AWS: adapters/aws/src/contractforge_aws
  2. Adaptador Databricks: adapters/databricks/src/contractforge_databricks
  3. Adaptador Snowflake: adapters/snowflake/src/contractforge_snowflake
  4. Adaptador Fabric: adapters/fabric/src/contractforge_fabric
  5. Adaptador GCP: adapters/gcp/src/contractforge_gcp

Revise as regras​

  1. Revise cada adaptador de forma independente.
  2. Fundamente cada descoberta em um arquivo e número de linha.
  3. Prefer concrete risks over generic style advice.
  4. Trate o código de tempo de execução gerado como código de produção quando for executado por Glue, Serviço Databricks, Snowflake, Fabric ou GCP.
  5. Registre os controles positivos e as violações quando eles afetarem o risco.
  6. Não marque um item como violado apenas porque ele existe. Por exemplo, except Exception é aceitável apenas quando preserva o contexto, edita dados confidenciais e faz parte de um limite explícito do adaptador.
  7. Use o modelo de gravidade abaixo de forma consistente.

Modelo de gravidade​

LevelCriterion
CRITICALProblema de segurança explorável, provável perda de dados, corrupção silenciosa da produção ou operação destrutiva ilimitada.
HIGHViolação grave de arquitetura, governança/autenticação ignorável, ampla instabilidade de produção ou bloqueador de testes.
MEDIUMMás práticas recorrentes, implementação confusa, falta de proteção ou dívida técnica que pode se tornar risco operacional.
LOWMelhoria local, pequeno problema de estilo, lacuna na documentação ou otimização de baixo risco.

Padrão de Evidência​

Cada descoberta deve incluir:

  • nome do adaptador;
  • dimension;
  • severity;
  • checklist item;
  • caminho do arquivo e número da linha;
  • evidência de código curto;
  • production impact;
  • recommended fix.

Quando um item da lista de verificação for avaliado e nenhum problema for encontrado, não crie uma descoberta. Mencione controles positivos importantes no sumário executivo somente quando for útil.

1. Security​

Gerenciamento de credenciais e dados confidenciais​

  • [] Credenciais, tokens, chaves API, PATs, chaves privadas, senhas ou segredos são codificados no código do adaptador, comentários, documentos, modelos, testes destinados como exemplos de produção ou arquivos de configuração confirmados.
  • [] Dados confidenciais podem aparecer em logs, artefatos de revisão renderizados, comandos saída, tags de consulta, linhas de evidências, notas de exceção, rastreamentos de pilha ou código de tempo de execução gerado.
  • [] Variáveis ​​de ambiente são lidas sem padrões explícitos, validação ou clear failure messages.
  • [] Espaços reservados secretos não são aplicados para campos de autenticação do conector que podem carry credentials.
  • [] A redação secreta está faltando no tratamento de erros do adaptador, na gravação de evidências, caminhos de falha de tempo de execução ou saída CLI.
  • [] Arquivos de configuração de smoke local com material de conexão real são referenciados em documentos ou exemplos comprometidos sem aviso ou exclusão.

Validação e higienização de entradas​

  • [] Campos de contrato externo são aceitos sem tipo, tamanho, valor permitido ou validação de identificador.
  • [] Os campos críticos não possuem listas de permissões, incluindo nomes de tabelas, nomes de esquemas, warehouse names, role names, stage names, task names, job names, IAM/Lake Princípios de formação, objetos Unity Catalog e nomes de recursos GCP.
  • [] Cargas de terceiros são desserializadas sem validação de esquema ou explicit shape checks.
  • [] Strings de consulta, partes URL, fragmentos de caminho, chaves de armazenamento de objeto ou estágio caminhos são usados ​​diretamente sem normalização e verificações de segurança.
  • [] Dicionários de extensão específicos do adaptador são transmitidos sem validando chaves, modos e valores de política conhecidos.

Injection Vulnerabilities​

  • [] SQL é construído a partir de entrada externa sem citação de identificador, literal escapando ou um renderizador revisado.
  • [] Os comandos Shell são montados a partir de entrada externa ou passados shell=True, os.system ou equivalente.
  • [] Python, SQL, YAML, JSON ou modelos de tarefas gerados injetam dados externos sem escape ou validação de sintaxe.
  • [] Os caminhos dos arquivos são criados a partir da entrada do usuário sem verificar a passagem do caminho, caminhos absolutos, caminhos qualificados para unidade ou passagem de caminho de estágio.
  • [] eval, exec, pickle ou importações dinâmicas executam dados não confiáveis.
  • [] Os executores da biblioteca de tempo de execução executam o código gerado sem uma lista de permissões, passagem de validação ou limite de confiança explícito.

Dependências e bibliotecas​

  • [] As dependências do adaptador são declaradas sem limites de versão ou pinos que fazem releases reproducible.
  • [] Versões de dependências vulneráveis ​​conhecidas estão presentes no resolvido ambiente ou arquivo de bloqueio.
  • [] Extras de tempo de execução opcionais são importados no caminho de importação principal padrão.
  • Unused imports add unnecessary dependency surface.
  • [] Bibliotecas obsoletas ou não mantidas são usadas para caminhos de produção.

Rede e comunicação​

  • HTTP calls omit timeouts.
  • [] A validação do certificado TLS está desabilitada ou configurável para valores inseguros sem uma política explícita de exigência de revisão.
  • [] Dados confidenciais são enviados em parâmetros de consulta URL onde cabeçalhos ou segredos gerentes devem ser usados.
  • [] Clientes personalizados TLS, JDBC, ODBC, armazenamento de objetos ou clientes de plataforma omitem validação de certificado/host ou padrões seguros.
  • [] Paginação API externa, novas tentativas ou tratamento de falhas podem repetir para sempre ou silently drop partial results.

Controle de acesso e autenticação​

  • [] As operações do adaptador assumem funções administrativas amplas em vez de privilégios mínimos funções de execução.
  • [] Concessão/revogação destrutiva, eliminação de tabela, exclusão de tarefa/procedimento ou política as alterações podem ser executadas sem aceitação explícita ou controle obrigatório de revisão.
  • [] Os tokens são armazenados em cache sem expiração, renovação ou separação de escopo.
  • [] As operações confidenciais não possuem limitação de taxa, pesquisa limitada ou limites de repetição.
  • [] O acesso entre contas, entre espaços de trabalho, entre catálogos ou entre projetos é não explicitado no ambiente ou nos artefatos de implantação.

2. Development Standards​

Python Idioms​

  • [] Usa antipadrões Python, como type(x) == T, == None, mutável padrões, ampla veracidade onde verificações explícitas são necessárias ou manual limpeza de recursos onde os gerenciadores de contexto são apropriados.
  • [] Os recursos que requerem limpeza não são gerenciados com with, try/finally ou semântica de fechamento específica do adaptador.
  • [] Loops detalhados poderiam ser mais claros como compreensões sem prejudicar readability.
  • [] Usa range(len(x)) onde enumerate é mais claro.
  • [] A formatação da string é inconsistente ou insegura para o idioma de destino.
  • [] Módulos API públicos não possuem __all__ onde o adaptador exporta intencionalmente uma superfície estável.

Digite dicas e contratos​

  • [] Funções ou métodos públicos não têm dicas de parâmetro ou tipo de retorno.
  • [] Any é usado onde um protocolo, dataclass, TypedDict, Mapping, Sequence ou modelo de domínio descreveria melhor o contrato.
  • [] As dicas de tipo discordam do comportamento do tempo de execução.
  • [] Dicionários de retorno complexos não possuem classes de dados, TypedDicts ou documentados payload schemas.
  • [] As funções podem retornar None implicitamente, mas não declaram esse comportamento.

Padrões de design e arquitetura​

  • [] Uma classe ou função viola a responsabilidade única ao misturar planejamento, renderização, implantação, execução em tempo de execução, gravação de evidências e limpeza em one unit.
  • [] Clientes de plataformas concretas são instanciados dentro da lógica de negócios sem caminho de injeção para teste.
  • [] Os limites do adaptador não possuem protocolos ou pequenas abstrações de wrapper para clientes da plataforma.
  • [] A herança é usada onde os registros de composição ou estratégia seriam simpler.
  • [] Módulos de Deus, funções de Deus ou classes de Deus possuem comportamento não relacionado.
  • [] A lógica de negócios é misturada com a mutação da infraestrutura de uma forma que bloqueia simulação, somente revisão ou testes de unidade.

Documentação​

  • [] Funções públicas ou comandos CLI não possuem docstrings ou documentação de uso onde o comportamento não é óbvio.
  • [] Documentos, exemplos de README ou documentos de sites estão desatualizados em relação aos atuais comportamento.
  • Comments explain obvious code rather than non-obvious decisions.
  • [] O comportamento exigido pela revisão é implementado, mas não documentado.
  • [] Pré-requisitos, subsídios, funções, comportamento de limpeza e custo para fumar ao vivo implicações estão ausentes ou são ambíguas.

PEP 8 e estilo local​

  • [] Linhas excedem a tolerância do projeto local sem legibilidade justification.
  • [] Os nomes não seguem as convenções Python ou convenções de adaptadores locais.
  • [] As importações não são ordenadas ou não são agrupadas como stdlib, de terceiros, local.
  • Top-level functions/classes have inconsistent spacing.
  • [] O novo código não segue o layout, registro ou renderização existente do adaptador conventions.

3. Code Reuse​

Internal Duplication​

  • [] Renderização, validação, redação, pesquisa, paginação, citação ou a lógica de gravação de evidências é duplicada dentro do mesmo adaptador.
  • [] A lógica de transformação de dados é copiada entre caminhos de tempo de execução sem um compartilhamento helper.
  • [] As mesmas regras de validação são reescritas em vários pontos de entrada.
  • [] Literais repetidos ou nomes de tabelas de controle devem ser nomeados como constantes.

Estruturação e Extração​

  • [] Funções longas com mais de 30 linhas devem ser decompostas, a menos que sejam simples declarative renderers.
  • [] Funções com mais de 50 linhas precisam de justificativa ou extração explícita.
  • [] A lógica in-line complexa deve ser extraída em auxiliares privados nomeados.
  • [] O aninhamento profundo deve ser substituído por cláusulas de guarda ou retornos antecipados.
  • [] Cadeias if/elif semelhantes a switch devem se tornar registros ou mapas de despacho quando eles aumentam ou espelham tabelas de capacidade do adaptador.

Existing Abstractions​

  • Stdlib utilities are reimplemented unnecessarily.
  • [] Ajudantes principais do projeto para redação, nomenclatura, validação, manipulação de fontes, política de esquema ou registros de evidências são ignorados.
  • [] Os padrões de adaptador existentes são ignorados sem um motivo documentado.
  • [] Constantes compartilhadas são copiadas em vez de importadas do local do adaptador constants module.

4. Qualidade do código​

Readability​

  • Variable names are semantically weak outside trivial loops.
  • [] As funções levam mais de 3-4 parâmetros relacionados onde um dataclass/config object would improve clarity.
  • [] As condições booleanas são desnecessariamente invertidas ou duplamente negadas.
  • [] A lógica condicional tem mais de 3 níveis aninhados sem extração auxiliar.
  • [] Números mágicos ou strings mágicas são usados ​​sem constantes nomeadas.

Exception Handling​

  • [] except Exception ou except simples captura falhas inesperadas sem preservando o contexto, redigindo segredos ou marcando o caminho como diagnóstico.
  • [] As exceções são capturadas e descartadas com pass sem motivo de recuperação.
  • Re-raised exceptions lose original context.
  • [] Exceções são usadas como fluxo de controle normal onde verificações explícitas são available.
  • [] Os erros de domínio do adaptador não possuem tipos de exceção personalizados ou mensagens de erro claras.
  • [] As mensagens de erro não possuem contexto suficiente para diagnósticos de produção.

Complexity​

  • [] A alta complexidade ciclomática cria muitos caminhos de execução independentes.
  • [] As funções excedem 50 linhas sem decomposição clara.
  • [] As classes expõem mais de 10 métodos públicos sem coesão clara.
  • [] Os módulos excedem 300 linhas e misturam preocupações não relacionadas.
  • [] Os renderizadores de código gerado são difíceis de inspecionar, validar ou testar.

Consistency​

  • [] Funções semelhantes retornam tipos ou formas de carga inconsistentes.
  • [] Operações semelhantes tratam erros de maneira diferente, sem motivo.
  • [] Os estilos de registro e saída CLI são inconsistentes.
  • [] Parâmetros semelhantes usam nomes diferentes no planejador, tempo de execução, CLI e smoke paths.
  • [] As semânticas de simulação, somente validação, execução, espera e limpeza diferem entre comandos semelhantes sem documentação.

5. Performance​

E/S e chamadas externas​

  • [] Chamadas externas API acontecem dentro de loops onde lote ou paginação podem ser used.
  • [] HTTP reutilizado, banco de dados, Spark, Snowflake, Databricks, AWS, Fabric ou GCP os clientes não são agrupados ou injetados.
  • [] Arquivos grandes ou artefatos de stage são lidos inteiramente na memória onde streaming would be practical.
  • [] As chamadas de bloqueio são usadas onde o adaptador já possui compatibilidade assíncrona infrastructure.
  • [] As chamadas de E/S não têm tempo limite, pesquisa limitada ou caminhos de cancelamento.

Estruturas de dados e algoritmos​

  • [] As listas são usadas para verificações repetidas de membros, onde os conjuntos são mais claros e faster.
  • [] Grandes estruturas são recriadas em cada chamada sem necessidade.
  • [] Os dados são repetidamente classificados ou normalizados sem armazenamento em cache dentro do mesmo flow.
  • [] A concatenação de strings em loops deve usar join.

Cache e otimização​

  • [] Operações determinísticas caras não são armazenadas em cache onde chamadas repetidas são esperado.
  • [] A configuração estática ou os metadados da plataforma são recarregados repetidamente em um invocação de adaptador único.
  • Idempotent computations are repeated multiple times in one flow.
  • [] Os intervalos de pesquisa ou esperas máximas são ilimitados ou não configuráveis.

Memory​

  • Large objects are retained beyond their useful scope.
  • [] Listas ansiosas devem ser iteradores/geradores preguiçosos para grandes conjuntos de dados.
  • [] Classes instanciadas muitas vezes com atributos fixos podem se beneficiar de dataclasses com slots ou __slots__.

6. Testability​

Acoplamento e injeção de dependência​

  • [] Clientes de plataforma, clientes HTTP, sessões de banco de dados, acesso ao sistema de arquivos ou relógio as funções são instanciadas internamente sem ponto de injeção.
  • [] Variáveis ​​globais ou singletons compartilham estado mutável entre testes ou execuções.
  • [] As importações em nível de módulo têm efeitos colaterais ou requerem tempo de execução opcional dependencies on import.
  • [] Chamadas para datetime.now, uuid.uuid4 ou geradores aleatórios afetam os testes sem injeção, ajudantes de wrapper ou asserções estáveis.

Side Effects​

  • [] Funções lidas do sistema de arquivos, rede, APIs de nuvem ou sessões de plataforma sem uma abstração intermediária ridícula.
  • [] Funções misturam lógica pura de planejamento/renderização com mutações externas.
  • [] Funções mutantes não retornam dados estruturados suficientes para asserções.
  • [] Os caminhos de limpeza não são executados em blocos finally para smoke ativa ou destructive tests.

Coverage Expectations​

  • Error paths lack apparent tests.
  • [] Ramos críticos carecem de testes de sucesso e fracasso.
  • [] Entradas vazias, nulas, malformadas e de tamanho máximo não são cobertas onde affect safety.
  • [] Tempo limite, serviço indisponível, permissão negada e falha parcial comportamento carece de testes.
  • [] O comportamento de fumar apenas ao vivo carece de uma aproximação testável por unidade.

Test Structure​

  • [] As classes não podem ser instanciadas isoladamente para testes de unidade.
  • [] Luminárias óbvias estão faltando ou duplicadas.
  • [] Os métodos têm múltiplas responsabilidades que forçam a configuração de testes pesados.
  • [] Funções void devem retornar resultados estruturados para apoiar afirmações.
  • [] Os comandos CLI não podem ser testados sem credenciais reais de nuvem.

Lista de Execução​

Execute esta lista para cada adaptador de forma independente. Preencha um relatório do adaptador antes iniciando o próximo relatório do adaptador.

1. Prepare Review Context​

  • [] Identifique a raiz do pacote do adaptador.
  • [] Identifique os pontos de entrada do adaptador CLI.
  • [] Identifique as dependências de tempo de execução opcionais do adaptador.
  • [] Identifique documentos de adaptadores, comandos de smoke e matrizes de estabilização.
  • [] Identifique testes de adaptadores e testes de smoke viva.
  • Confirm whether live credentials are available.
  • Confirm whether dependency audit tooling is available.

2. Run Static Discovery​

  • [] Pesquise credenciais codificadas, tokens, senhas, chaves privadas, PATs, Chaves API e arquivos de conexão confirmados.
  • [] Pesquise caminhos de redação em torno de logs, erros, evidências, saída CLI e código de tempo de execução gerado.
  • [] Pesquisa por execução dinâmica insegura: eval, exec, pickle, dinâmica importações, execução Python gerada em tempo de execução e executores de biblioteca.
  • [] Pesquise a execução do shell: subprocess, os.system, strings do shell e wrappers de comando da plataforma.
  • [] Pesquise a construção SQL e verifique o identificador/escape literal.
  • [] Pesquise a construção do caminho e verifique as proteções de passagem.
  • [] Pesquise chamadas HTTP/JDBC/plataforma e verifique tempos limite, novas tentativas e TLS defaults.
  • [] Pesquise um tratamento amplo de except Exception e verifique o contexto/redação.
  • [] Pesquise importações em nível de módulo de SDKs de nuvem opcionais na importação padrão paths.
  • [] Pesquise módulos grandes e funções longas que possam precisar de decomposição.

3. Review Security​

  • [] Avalie o gerenciamento de credenciais e dados confidenciais.
  • [] Avaliar validação e higienização de insumos.
  • [] Avalie os riscos de SQL, shell, modelo, caminho e injeção de execução dinâmica.
  • [] Avalie a dependência e o isolamento extra opcional.
  • [] Avalie a segurança da rede: TLS, tempo limite, nova tentativa e comportamento de paginação.
  • [] Avalie o controle de acesso, controle de operação destrutiva e privilégio mínimo.
  • [] Registre apenas descobertas concretas com evidências de arquivo/linha.

4. Review Development Standards​

  • [] Avalie expressões idiomáticas Python e padrões de limpeza de recursos.
  • [] Avalie dicas de tipo e contratos.
  • [] Avalie a arquitetura do adaptador e a separação de planejamento, renderização, questões de tempo de execução, implantação, evidências e limpeza.
  • [] Avalie a documentação e os exemplos em relação ao comportamento real.
  • [] Avalie o PEP 8 e a consistência do estilo do adaptador local.
  • [] Registre apenas descobertas concretas com evidências de arquivo/linha.

5. Review Code Reuse​

  • Assess duplicated validations, renderers, redaction, quoting, polling, paginação e lógica de escrita de evidências.
  • [] Avalie funções longas e lógica inline complexa para extração.
  • [] Avalie se os auxiliares principais e as abstrações locais do adaptador são reutilizados.
  • [] Registre apenas descobertas concretas com evidências de arquivo/linha.

6. Revise a qualidade do código​

  • [] Avalie a legibilidade, formato dos parâmetros, nomenclatura, booleanos e valores mágicos.
  • [] Avalie o tratamento de exceções e a clareza dos erros no domínio do adaptador.
  • [] Avalie a complexidade por função, classe e módulo.
  • [] Avalie a consistência entre CLI, planejador, tempo de execução, implantação, smoke e documentos.
  • [] Registre apenas descobertas concretas com evidências de arquivo/linha.

7. Review Performance​

  • [] Avalie chamadas externas em loops e comportamento de paginação.
  • [] Avaliar a reutilização e injeção de cliente/sessão.
  • [] Avalie o comportamento de arquivos grandes e memória de artefatos.
  • [] Avalie escolhas de estrutura de dados e cálculos determinísticos repetidos.
  • [] Avalie oportunidades de cache e pesquisas limitadas.
  • [] Registre apenas descobertas concretas com evidências de arquivo/linha.

8. Review Testability​

  • [] Avaliar injeção de dependência para clientes de plataforma, sistema de arquivos, relógios, UUIDs, valores aleatórios e sessões ao vivo.
  • [] Avalie a separação entre lógica pura de planejamento/renderização e efeitos colaterais.
  • [] Avaliar caminho de erro, tempo limite, permissão negada, entrada malformada e limpeza test coverage.
  • [] Avalie se os comandos CLI podem ser testados sem credenciais reais de nuvem.
  • [] Registre apenas descobertas concretas com evidências de arquivo/linha.

9. Run Verification​

  • [] Execute testes de unidade com foco no adaptador.
  • [] Execute testes entre adaptadores afetados pelo adaptador.
  • [] Execute o conjunto completo de testes quando alterações ou descobertas afetarem contratos compartilhados.
  • [] Execute verificações de compilação/pacote.
  • [] Execute verificações de ajuda do CLI para pontos de entrada do adaptador.
  • [] Execute smoke ativa somente quando as credenciais estiverem disponíveis e a limpeza for segura.
  • [] Verifique se a limpeza de smoke ao vivo não deixa recursos temporários criados pelo adaptador.

10. Produzir relatório do adaptador​

  • [] Escreva o relatório do adaptador usando o modelo de relatório.
  • Include severity metrics.
  • Include health score.
  • Include top 3 immediate actions.
  • [] Incluir controles positivos importantes no sumário executivo quando eles reduce risk.
  • [] Evite descobertas sem remediação acionável.

11. Produce Consolidated Report​

  • [] Adicionar tabela comparativa de pontuação de saúde.
  • [] Adicione padrões recorrentes entre adaptadores.
  • Add global risk/effort prioritization matrix.
  • [] Separe as descobertas específicas do adaptador das recomendações entre adaptadores.
