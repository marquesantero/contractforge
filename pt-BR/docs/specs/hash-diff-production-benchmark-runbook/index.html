Hash-Diff Production Benchmark Runbook

Propósito​

Este runbook define o benchmark compartilhado necessário antes de AWS, GCP e Snowflake pode fechar suas portas de certificação de produção hash_diff_upsert.

Não é um teste de marketing de desempenho. O objetivo é provar o comportamento limitado: detecção correta de linha alterada, reprodução segura, simultaneidade segura ou sobreposição manuseio, evidência de falha redigida e atribuição de custos.

Os manifestos de benchmark específicos do adaptador são:

  • AWS hash-diff production benchmark
  • GCP BigQuery hash-diff production benchmark
  • Snowflake hash-diff production benchmark

O fixture executável compartilhado é:

  • examples/benchmarks/hash-diff-production/project.yaml

Dataset Shape​

Use o mesmo esquema de origem lógica em ambas as plataformas:

ColunaTipoPropósito
customer_idinteger/stringChave de merge.
segmentstringDimensão de mudança de baixa cardinalidade.
statusstringDimensão de mudança de cardinalidade média.
balancedecimalDimensão de mudança numérica.
updated_attimestampDimensão de encomenda e auditoria.
payload_hash_noisestringCampo não chave para aumentar a largura da linha.

Tamanhos recomendados de benchmark:

SizeLinhasPropósito
small10,000Correção rápida e verificação do caminho de falha.
medium1,000,000Execução rotineira de certificação de produção.
large10,000,000+Verificação opcional de capacidade e envelope de custos.

Casos obrigatórios​

CaseResultado esperado
initial_loadO destino recebe todas as linhas de origem e registra evidências de execução/custo bem-sucedidas.
no_change_replayO adaptador não registra nenhuma linha duplicada e nenhuma linha alterada ou uma execução sem operação.
changed_row_waveSomente as linhas alteradas são mescladas; linhas inalteradas permanecem intactas.
concurrent_or_overlap_guardOs gravadores simultâneos serializam com segurança ou uma execução falha com evidências editadas.
duplicate_key_failureAs chaves de mesclagem duplicadas falham antes de gravar e registrar evidências redigidas de falha na execução/erro.
null_key_failureAs chaves de mesclagem nulas falham antes de gravar e registrar evidências redigidas de falha na execução/erro.

Métricas Obrigatórias​

Cada registro de benchmark ao vivo deve incluir:

  • adaptador e subalvo;
  • versão do pacote e git commit;
  • tamanho do conjunto de dados e contagem de linhas alteradas;
  • ID de execução ou ID de trabalho/consulta nativo;
  • elapsed seconds;
  • linhas lidas, linhas escritas e linhas candidatas a hash-diff;
  • linhas de sinal de custo;
  • versão da tabela ou estado pós-gravação equivalente;
  • evidência de falha para casos negativos;
  • cleanup status.

Padrão de comando AWS​

contractforge-aws deploy-project <benchmark-project.yaml> \
  --run \
  --wait \
  --record-cost-evidence \
  --audit-evidence \
  --athena-output-location <s3-query-output>

contractforge-aws performance-report <hash-diff-contract.yaml> \
  --environment <aws.environment.yaml> \
  --run \
  --athena-output-location <s3-query-output>

A evidência AWS deve incluir IDs de execução de trabalho Glue, linhas de auditoria do Athena, snapshot Iceberg métricas e linhas ctrl_ingestion_cost com glue_dpu_seconds.

Padrão de comando GCP​

contractforge-gcp smoke <hash-diff-contract.yaml> \
  --environment <gcp.environment.yaml> \
  --execute \
  --allow-review-required \
  --runtime bq

A evidência GCP deve incluir IDs de tarefas BigQuery, contagens de linhas DML, bytes processados, bytes faturados, milissegundos de slot, evidência de falha na execução para casos negativos e um readback provando que não há linhas duplicadas após repetição ou execução simultânea.

Padrão de comando Snowflake​

contractforge-snowflake run-project <benchmark-project.yaml> \
  --connect-options <connection.yaml> \
  --wait

contractforge-snowflake cost-report \
  --connect-options <connection.yaml> \
  --environment <environment.json> \
  --run-id <contractforge-run-id> \
  --target-table <qualified-target-table> \
  --wait \
  --max-wait-seconds 3600

A evidência Snowflake deve incluir IDs de consulta, linhas da tabela de controle, QUERY_HISTORY/sinais de custo de atribuição de consulta quando o uso da conta é detectado up e IDs de tarefas/procedimentos quando o caminho do projeto é usado.

Closure Rule​

O manifesto de benchmark específico do adaptador pode passar de READY_TO_RUN para PASS only after:

  1. todos os casos exigidos possuem evidências reais;
  2. evidências de custo e estado estão presentes;
  3. casos de falha mostram evidências de erro redigidas;
  4. cleanup is verified;
  5. o portão do rastreador de maturidade correspondente é atualizado.
