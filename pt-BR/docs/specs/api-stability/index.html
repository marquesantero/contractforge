Estabilidade e controle de versão do API

Propósito​

ContractForge publica o núcleo, os adaptadores e o complemento de IA como Python separado distribuições. Este documento define o que cada pacote promete como API público, o que resta da visualização e como as versões dos pacotes se movem de forma independente.

Política de Versionamento​

Os pacotes ContractForge usam versionamento semântico independente. As versões do pacote fazem não se mova em sincronia.

O primeiro lançamento público mantém as versões já presentes no repositório:

DistributionFirst public versionStability
contractforge-core0.1.0Limite API público beta.
contractforge-databricks0.1.0Superfície do adaptador alfa.
contractforge-aws0.1.0Superfície suportada estável para aws_glue_iceberg; os limites avançados permanecem como visualização.
contractforge-snowflake0.1.0Superfície suportada estável para snowflake_sql_warehouse; os limites avançados permanecem como visualização.
contractforge-ai0.2.9Alpha AI companion surface.

A compatibilidade é declarada através do campo dependencies de cada pacote. Há não há promessa de que dois pacotes com o mesmo número de versão sejam lançamentos relacionados.

Public API​

Para contractforge-core, API público inclui módulos importáveis ​​em contractforge_core cujos segmentos do caminho do módulo não começam com _, mais o documented contractforge CLI commands.

O API público principal inclui:

  • modelos de contrato semântico
  • análise de projeto e ajudantes de cronograma
  • modelos de capacidade e correspondência
  • metadados do catálogo de conectores
  • tipos de resultados de planejamento
  • modelos de redação de evidências e segurança
  • comportamento documentado do CLI

Os módulos principais ou nomes prefixados com _ são internos. Futuro _internal os pacotes também são internos.

Preview API​

Os pacotes de adaptadores são alfa até serem explicitamente promovidos:

  • contractforge-databricks

Adaptadores Alpha podem alterar formatos de artefatos específicos do adaptador, auxiliar de tempo de execução APIs, campos de ambiente e cargas de extensão em versões secundárias. Eles devem ainda preservam a direção de dependência do adaptador principal e devem declarar seus compatible contractforge-core range.

contractforge-aws é promovido para o estável aws_glue_iceberg documentado superfície apoiada. Seus limites restantes de certificação de produção, incluindo provedores de streaming genéricos, governança específica de carga de trabalho/declarações SLA e semântica de exclusão reversível de histórico/instantâneo excluída, permanece com visualização ou revisão obrigatória until their criteria are met.

contractforge-snowflake é promovido para o documentado Superfície suportada estável snowflake_sql_warehouse. Seu restante limites de certificação de produção, incluindo ingestão contínua, validação de política de acesso dependente de recursos da conta e histórico/instantâneo excluído semântica de exclusão reversível, permanecem com visualização ou revisão obrigatória até que seus critérios are met.

contractforge-ai é alfa. Estrutura de projeto gerada, roteamento de provedor relatórios, modelos de prompt e cargas de enriquecimento podem mudar em versões secundárias. O limite de validação determinístico permanece oficial: resultados gerados devem passar pela validação principal antes de serem tratados como ContractForge utilizáveis projects.

Internal API​

Os itens a seguir são internos e podem ser alterados sem depreciação:

  • módulos, pacotes, funções, classes ou atributos prefixados com _
  • código de tempo de execução gerado que não está documentado como auxiliar público
  • acessórios de teste e artefatos .tmp
  • prompt interno do provedor, a menos que documentado nos documentos do pacote AI
  • detalhes de implementação do adaptador não exportados do pacote __init__.py

Breaking Changes​

Before 1.0.0, minor versions may include breaking changes in preview superfícies. As versões de patch devem permanecer apenas para correção de bugs.

Após 1.0.0, alterações significativas no API público estável exigem uma versão principal aumentar. Adaptadores Alpha e AI podem permanecer abaixo de 1.0.0 enquanto o núcleo atinge 1.0.0.

Política Databricks GA​

Quando o adaptador Databricks atinge GA, contractforge-databricks pode passar para 1.0.0. O núcleo se move para 1.0.0 com ele somente se o limite GA depender de um contrato API público central estável. AWS, Snowflake e AI continuam por conta própria version tracks.

STABLE_SUPPORTED_SURFACE é uma classificação mais restrita de preparação para liberação para a superfície documentada do adaptador v0.1.0. Pode ser stable_final=true enquanto o O portão Databricks 1.0.0 GA, as matrizes completas do provedor e as decisões de congelamento do API permanecem separate.
