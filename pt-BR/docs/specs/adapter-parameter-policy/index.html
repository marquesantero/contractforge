Política de parâmetros do adaptador

Propósito​

Esta política define como o ContractForge Core nomeia os parâmetros do contrato e como os adaptadores de plataforma adicionam comportamento nativo sem forçar o contrato principal a espelhar uma plataforma.

O núcleo possui nomes de parâmetros semânticos. Tradução própria dos adaptadores.

Parameter Layers​

Os parâmetros ContractForge são classificados em quatro camadas.

LayerOwnerPropósitoExemplo
Semântica centralEssencialIntenção neutra em termos de plataforma.source.type: incremental_files, progress_location, schema_policy
Detalhe opcional principalEssencialDetalhe extra portátil que várias plataformas podem usar.schema_tracking_location, watermark, merge_keys
Parâmetro do adaptadorAdaptadorAjuste nativo ou opções nativas necessárias.environment.parameters.databricks, environment.parameters.aws
Artefato nativoAdaptadorRecurso de plataforma renderizado, não entrada de contrato.Databricks Asset Bundle, Glue job script, Fabric pipeline JSON

Os parâmetros do adaptador residem em environment.parameters.<adapter>.

Regras de nomenclatura​

Os nomes principais devem descrever a intenção, não uma implementação.

Exemplos preferidos:

PreferEvite no núcleoReason
incremental_filesautoloaderAuto Loader is Databricks-specific.
progress_locationcheckpoint_locationCheckpoint is implementation vocabulary; progress is semantic.
schema_tracking_locationschema_locationO rastreamento de esquema descreve a intenção entre os mecanismos.
native_passthroughlakeflow_connect / appflowO serviço de conector nativo difere de acordo com a plataforma.
access.row_filtersunity_catalog_row_filtersA filtragem de linhas é uma intenção de plataforma cruzada.
annotations.tagsuc_tags / lf_tags / policy_tagsTag implementation differs by catalog.

Os nomes dos adaptadores Databricks não devem se tornar aliases de compatibilidade no núcleo. Os nomes específicos da plataforma devem ser convertidos antes da validação, em vez de serem aceitos pelos APIs de tempo de execução.

Sem aliases principais​

O núcleo não preserva aliases específicos da plataforma.

Não permitido no núcleo:

  • source.type: autoloader
  • checkpoint_location
  • schema_location
  • glue_bookmark
  • lakeflow_connect_*
  • appflow_*
  • dms_*
  • dataflow_gen2_*
  • shortcut_*

Contratos de tempo de execução externos permitidos:

  • auxiliares de conversão que reescrevem nomes específicos da plataforma em nomes canônicos antes da validação
  • diagnósticos que documentam por que um alias nativo foi rejeitado
  • artefatos renderizados usando nomes de plataforma nativos
  • documentos do adaptador explicando como os campos principais canônicos são mapeados para parâmetros nativos

Extension Shape​

Os parâmetros portáteis permanecem na seção normal do contrato.

Os parâmetros específicos do adaptador residem em environment.parameters.<adapter>.

Recommended shape:

source:
  type: incremental_files
  path: s3://bucket/landing/events/
  format: json
  progress_location: s3://bucket/_progress/events/
  schema_tracking_location: s3://bucket/_schemas/events/
environment:
  name: prod
  adapter: databricks
  parameters:
    databricks:
      incremental_files.infer_column_types: true
      job.warehouse_id: ${DATABRICKS_WAREHOUSE_ID}

Regras:

  • environment.parameters.<adapter> configura preocupações de implantação/renderização de propriedade do adaptador e padrões de plataforma.
  • os adaptadores devem ignorar blocos de extensão para outras plataformas.
  • os adaptadores devem avisar quando uma extensão nativa necessária está faltando e não pode ser padronizada com segurança.

Quando uma plataforma precisa de mais parâmetros​

Os adaptadores devem seguir esta ordem:

  1. Use o parâmetro semântico principal.
  2. Use um padrão de plataforma seguro, se existir, e documente-o como um aviso quando relevante.
  3. Leia os parâmetros específicos do adaptador em environment.parameters.<adapter>.
  4. Return REVIEW_REQUIRED if a design choice is needed.
  5. Return UNSUPPORTED if no safe equivalent exists.

Exemplos:

ContractForge intentNecessidade de plataformaComportamento do adaptador
incremental_filesDatabricks needs cloudFiles.schemaLocation.Utilize schema_tracking_location; se estiver faltando, use aviso de revisão ou exija extensão dependendo do modo de execução.
incremental_filesAWS Glue may need bookmark key/state configuration.Use progress_location se o estado externo for renderizado; caso contrário, leia os padrões environment.parameters.aws.
annotations.columns.email.piiGCP pode precisar do ID de taxonomia da tag de política.Read environment.parameters.gcp.policy_tags; otherwise REVIEW_REQUIRED.
access.column_masks.email.functionSnowflake pode precisar de assinatura de política de mascaramento.Use a função se for compatível; caso contrário, exigirá environment.parameters.snowflake.masking_policy.
operations.alert_on_failureAWS precisa do destino SNS/EventBridge.Read environment.parameters.aws.alerting; otherwise REVIEW_REQUIRED.

Quando renomear um parâmetro​

Renomeie um parâmetro principal quando:

  • o nome está vinculado ao produto de uma plataforma ou API
  • pelo menos duas plataformas podem implementar a mesma intenção sob nomes diferentes
  • o nome atual esconde uma distinção semântica
  • exemplos levariam os usuários a pensar que um contrato portátil é específico da plataforma

Não renomeie quando:

  • o parâmetro já é semântico e legível
  • apenas uma plataforma pode implementar o conceito
  • o conceito deveria ser uma extensão da plataforma

Saída de planejamento do adaptador​

Os adaptadores devem relatar decisões em nível de parâmetro no planejamento de diagnósticos quando o comportamento não for totalmente direto.

O diagnóstico deve incluir:

  • caminho do parâmetro do contrato
  • alvo do adaptador
  • status
  • mapeamento nativo
  • faltando parâmetros nativos, se houver
  • motivo de aviso ou revisão

Exemplo:

{
  "parameter": "source.schema_tracking_location",
  "adapter": "databricks",
  "status": "SUPPORTED",
  "native_mapping": "cloudFiles.schemaLocation"
}

Regra Básica​

O núcleo aceita blocos de extensão do adaptador como dados opacos. Ele não deve validar, normalizar, renomear ou interpretar chaves de propriedade do adaptador.

O adaptador possui artefatos de execução nativos e semântica de opções nativas.
