Especificação de portabilidade de origem

Propósito​

O ContractForge Core classifica os tipos de origem por portabilidade antes que um adaptador tente planejar ou renderizar a execução.

O contrato de origem deve tornar o limite explícito:

  • fonte interna portátil
  • fonte de passagem nativa
  • fonte de fluxo limitada
  • fonte de transmissão disponível agora
  • fonte não suportada ou não portátil

Isso evita que conectores específicos da plataforma vazem para o núcleo semântico, ao mesmo tempo que permite que os adaptadores usem o melhor recurso nativo disponível.

Portable Built-Ins​

Esses tipos de origem são mantidos como intenção portátil de nível central:

CategoryTipos de fonte
Lakehouse catalogtable, delta_table, iceberg_table, view, sql
File formatscsv, json, parquet, delta, orc, text, avro, xml
Object storages3, adls, azure_blob, gcs, blob, object_storage
Incremental filesincremental_files
HTTP fetchhttp_file, http_csv, http_json, http_text
JDBC batchjdbc, postgres, mysql, sqlserver, oracle, redshift, db2, mariadb
Bounded streamskafka_bounded, eventhubs_bounded
Available-now streamskafka_available_now, eventhubs_available_now
Lakehouse sharingdelta_share
Connection referenceconnection

oracle é uma intenção JDBC portátil, mas os adaptadores devem documentar que o usuário fornece o driver Oracle JDBC. O projeto não deve redistribuir ojdbc.

xml é uma intenção de arquivo portátil, mas as opções do analisador, como tags de linha, validação de esquema e manipulação de registros malformados, variam de acordo com o adaptador. Os contratos devem declarar as opções de analisador obrigatórias no source.options, e os adaptadores devem retornar avisos ou resultados de revisão obrigatória quando seu tempo de execução não puder preservar a semântica declarada do XML.

connection não é um conector de tempo de execução. É uma referência YAML reutilizável resolvida por load_contract_bundle() antes da normalização semântica. Após a resolução, os adaptadores veem a fonte concreta declarada pela conexão YAML, como connector/postgres, http_file, object_storage ou rest_api.

source:
  type: connection
  connection_path: project://connections/supabase.yaml
  table: public.products

Os caminhos project:// são resolvidos a partir da raiz project.yaml mais próxima. Mesmo pacote caminhos relativos são permitidos somente quando permanecem no pacote de ingestão diretório e não contém ... Caminhos absolutos são rejeitados. Específico do conjunto de dados campos no contrato de ingestão substituem os padrões de conexão por meio de merge.

As fontes de catálogo também podem usar referências de tabelas lógicas ao ler uma tabela criado por outro contrato ContractForge:

source:
  type: table
  ref: bronze.b_products_jdbc

As fontes SQL podem fazer referência à mesma tabela lógica por meio de espaços reservados:

FROM {{ table_ref:silver.s_product_tags }}

O núcleo valida a referência neutra layer.table. Adaptadores resolvem isso para nomes de tabelas nativas. Isso mantém os contratos medalhão posteriores portáteis sem ensinando as regras básicas de nomenclatura Unity Catalog, Glue Catálogo, Snowflake ou Fabric.

Intenção, descoberta e estado da fonte​

source.type continua sendo a família de fontes portáteis concretas, como s3, jdbc, incremental_files, rest_api ou native_passthrough.

Os contratos também podem declarar metadados de intenção opcionais quando a família de origem por si só não for suficiente para expressar a intenção de planejamento do adaptador:

  • source.intent: intenção de ingestão de alto nível, como file_stream, file_batch, database_query, catalog_query, api_call, object_files, stream_replay ou native_handoff.
  • source.discovery: como novos dados de origem são descobertos, com valores strategy, como file_listing, event_driven ou queue_based, e valores tracking, como modification_time, filename_pattern, external_state ou external_queue.
  • source.state: onde o progresso da origem é rastreado quando o contrato exige uma localização de estado explícita. storage: external requer um location; adapter_managed permite que o adaptador escolha o mecanismo de estado nativo.

Esses campos não introduzem o comportamento de execução da plataforma no núcleo. Eles fornecem aos adaptadores informações de planejamento mais precisas, preservando a carga útil da fonte original em evidência.

Exemplo:

source:
  type: s3
  intent: file_stream
  path: s3://bucket/landing/events/
  format: json
  discovery:
    strategy: file_listing
    tracking: modification_time
  state:
    storage: external
    location:
      type: object_storage
      path: s3://bucket/_state/events/

Um adaptador pode mapear isso para o carregador automático, marcadores Glue, listagem de objetos mais estado de evidência, um padrão incremental nativo de pipeline ou REVIEW_REQUIRED. Não deve converter silenciosamente o contrato em uma leitura de lote limitada mais fraca.

Incremental Files​

incremental_files é a intenção de fonte portátil para leitura de novos arquivos em um caminho com ponto de verificação/rastreamento de progresso.

Exemplos de adaptadores:

  • Databricks: Auto Loader cloudFiles
  • AWS: marcadores Glue ou estado de trabalho equivalente
  • Fabric: fluxo de dados incremental nativo da plataforma ou comportamento de pipeline

Nem os adaptadores principais nem de plataforma devem aceitar aliases de origem específicos da plataforma, como autoloader, como entrada de contrato. As ferramentas de migração podem reescrever contratos antigos antes da validação, mas o contrato de tempo de execução permanece incremental_files.

Canonical portable fields:

  • path
  • format
  • progress_location
  • schema_tracking_location
  • schema_hints
  • options

Os padrões do adaptador nativo para manipulação incremental de arquivos pertencem ao environment.parameters.<adapter>, não ao contrato de origem.

Bounded Streams​

kafka_bounded e eventhubs_bounded significam atualização/reprodução em um intervalo limitado de deslocamento ou carimbo de data/hora.

Eles não significam streaming contínuo. O streaming contínuo requer um modelo de execução separado com pontos de verificação, marcas d'água, comportamento exatamente uma vez, contrapressão e semântica de recuperação.

Available-Now Streams​

kafka_available_now e eventhubs_available_now significam atualização de fluxo com ponto de verificação: o adaptador lê todos os eventos atualmente disponíveis, persiste o progresso do fluxo em source.checkpoint_location e, em seguida, termina.

Isto não é streaming contínuo. Os adaptadores podem mapeá-lo para gatilhos availableNow nativos, tarefas de atualização agendadas ou REVIEW_REQUIRED quando o ponto de verificação e a semântica de gravação não puderem ser preservados.

Canonical portable fields:

  • checkpoint_location
  • Kafka: bootstrap_servers mais topic, topics ou assign
  • Event Hubs: connection_string e event_hub_name
  • Limites de taxa de transferência de fluxo: declare source.limits.max_offsets_per_trigger para Kafka e source.limits.max_events_per_trigger para Event Hubs. Nível superior max_offsets_per_trigger / max_events_per_trigger também são aceitos pelo modelo de origem para declarações de conector de fluxo direto; se ambos os locais estiverem presente com valores diferentes, a validação/renderização deve falhar em vez de silently choosing one.

Non-Portable Sources​

O núcleo não mantém uma lista de nomes de fontes específicos da plataforma. Os tipos de origem desconhecidos não são tipos de origem principais portáteis.

Os adaptadores podem suportar nomes de origem locais do adaptador, mas esses nomes não devem ser incluídos no contractforge_core.

Passagem nativa​

native_passthrough é um tipo de fonte de primeira classe para SaaS, aplicativos corporativos e conectores proprietários onde a ingestão nativa da plataforma é preferível à manutenção de um conector personalizado.

Exemplos:

  • Salesforce
  • Workday
  • SAP
  • SharePoint
  • Google Drive
  • OData e SAP OData
  • MongoDB, Cosmos DB, Elastic/OpenSearch
  • SFTP, FTP, IMAP

Os adaptadores traduzem a passagem nativa em seu serviço nativo:

  • Databricks: Lakeflow Connect ou conexão/pipeline nativo do espaço de trabalho
  • AWS: conector AppFlow, DMS, Glue ou trabalho nativo da plataforma
  • Fabric: Dataflow Gen2 ou conector de plataforma

Exemplos de contratos​

Portable JDBC:

source:
  type: jdbc
  jdbc:
    url: jdbc:postgresql://host:5432/db
    table: public.accounts
  watermark:
    column: updated_at
    type: timestamp

Portable incremental files:

source:
  type: incremental_files
  path: s3://bucket/landing/events/
  format: json
  progress_location: s3://bucket/_progress/events/
  schema_tracking_location: s3://bucket/_schemas/events/

Fonte de armazenamento de objetos portátil com intenção explícita de fluxo de arquivos:

source:
  type: s3
  intent: file_stream
  path: s3://bucket/landing/events/
  format: json
  discovery:
    strategy: file_listing
    tracking: modification_time

Passagem nativa:

source:
  type: native_passthrough
  system: salesforce
  object: Account
  watermark:
    column: SystemModstamp
  auth:
    type: oauth2_jwt
    secret_scope: sf_prod
