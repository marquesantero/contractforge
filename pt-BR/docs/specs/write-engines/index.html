Especificação de motores de gravação

Propósito​

Um modo de gravação ContractForge descreve a intenção semântica. Um mecanismo de gravação descreve como um adaptador executa essa intenção em uma plataforma específica.

O núcleo possui modos de gravação como anexar, substituir, upsert, hash-diff upsert e controle de versão histórico. Os adaptadores possuem seleção de mecanismo nativo. O mecanismo selecionado deve ser registrado como evidência para que os operadores possam ver se uma execução usou comportamento de plataforma nativa ou um algoritmo gerenciado por ContractForge.

Modos de gravação principais​

Modo de contratoPortable semantics
appendAnexe linhas preparadas ao destino sem corresponder às linhas existentes.
overwriteSubstitua o conteúdo de destino de acordo com a semântica de substituição suportada pelo adaptador.
replace_partitionsReplace scoped partitions only. Requires explicit partition predicate support.
upsertMesclagem do estado atual codificada por merge_keys. Requer mesclagem ou semântica de atualização equivalente.
hash_diff_upsertInserir/atualizar somente quando um hash de linha for alterado. Requer cálculo de hash e mesclagem ou semântica equivalente.
historicalPreserve o histórico com marcadores de linha atual e colunas de validade. Requer uma revisão mais rigorosa em todas as plataformas.
snapshot_reconcile_soft_deleteComparação de instantâneos com marcação de exclusão reversível para registros ausentes. Revisão necessária em muitos mecanismos.

O planejador nunca deve substituir um modo solicitado por um modo mais fraco. Por exemplo, historical não pode recorrer a append.

O mecanismo exato selecionado é persistido por execução. Os usuários devem ler as evidências em vez de inferir o comportamento de execução apenas do modo.

Request Syntax​

As solicitações de mecanismo específicas do adaptador residem em extensions.databricks.

extensions:
  databricks:
    write_engine:
      requested: lakeflow_auto_cdc
      fallback_policy: preview_only

Os valores fallback_policy suportados são de propriedade do adaptador. Databricks atualmente trata essas políticas como:

PolíticaComportamento
failFalha se o mecanismo solicitado não puder ser usado com segurança.
preview_onlyRenderize/revise o plano nativo, mas não execute silenciosamente um mecanismo alternativo como se fosse equivalente.
autoPermita que o adaptador escolha um mecanismo compatível quando nenhum mecanismo explícito for necessário.

Colunas de evidências​

Cada execução de produção deve preencher:

  • write_engine_requested: solicitação do usuário, geralmente auto quando não especificado.
  • write_engine_selected: motor concreto selecionado pelo adaptador.
  • write_engine_status: classificação como native_databricks, contractforge_algorithm, review_required ou unsupported.
  • write_engine_reason: breve seleção ou explicação do bloqueador.
  • write_engine_fallback_policy: política aplicada quando a solicitação e o mecanismo selecionado são diferentes.

Estas colunas pertencem ao modelo de evidência central. O adaptador os preenche com valores específicos da plataforma.

Requisitos de autoria do adaptador​

Cada adaptador deve definir:

  1. os modos de gravação suportados nativamente;
  2. os modos que suporta com um algoritmo gerenciado por ContractForge;
  3. os modos que requerem revisão;
  4. os modos que não são suportados;
  5. valores de evidência para solicitação, seleção, status, motivo e política de fallback.

Um adaptador pode ter menos mecanismos nativos que Databricks. Isso é aceitável se o planejador retornar o status correto e não alterar a semântica silenciosamente.

Quando uma plataforma precisa de mais parâmetros do que o modo principal expõe, o adaptador deve usar seu próprio namespace de extensão ou parâmetros de ambiente. Não deve adicionar campos específicos da plataforma à raiz do contrato principal.
