Matriz de Mapeamento de Evidências

Propósito​

Este é o mapeamento coluna por coluna do esquema canônico da tabela de controle ContractForge para a superfície de evidência nativa de cada plataforma do adaptador. É a referência oficial para "de onde vem o valor desta coluna na plataforma X" e o companheiro de control-table-parity.md (que define as colunas canônicas e o status de portabilidade) e evidence-model.md (que define o vocabulário de evidência portátil).

Platforms covered: Databricks, AWS (Glue + Iceberg), GCP (BigQuery / Dataproc), Snowflake, Microsoft Fabric.

Design Principles​

  1. Mesmas informações em todos os adaptadores. O esquema canônico é neutro em termos de plataforma e reside uma vez em contractforge_core.evidence. Cada adaptador preenche as mesmas colunas, portanto as evidências são comparáveis ​​entre plataformas.
  2. Imutável, somente acréscimo. As tabelas de evidências são somente anexadas (somente INSERT, particionadas por data) — imutáveis ​​como contratos. A tabela de estado é registrada como histórico somente de acréscimo; "estado atual" é a linha mais recente por destino. Nada é atualizado no local.
  3. JSON carrega a riqueza, nunca descartada. Qualquer coisa nativa que não tenha coluna de primeira classe é preservada em uma coluna *_json (operation_metrics_json, source_*_json, payload_json, details_json, event_json, batch_metrics_json). Relatórios/painéis futuros os exploram.
  4. Calcule, não diga "não posso". Quando a telemetria de execução de uma plataforma não expõe um contador diretamente, o adaptador o calcula no trabalho (ele controla o SQL/DataFrame). Exemplo: o resumo do instantâneo Iceberg não divide inserção versus atualização, mas o trabalho Glue MERGE conhece ambos - portanto, ele conta e preenche rows_inserted/rows_updated em vez de deixá-los nulos.
  5. Nomes neutros, valores nativos em JSON. As colunas são nomeadas por intenção, não pelo API de um fornecedor. Identificadores específicos do fornecedor (número de versão Delta, ID de instantâneo Iceberg, ID de trabalho BigQuery) são mapeados em uma coluna neutra e o valor nativo também é mantido em operation_metrics_json.

Immutability​

  • Tabelas de evidências (runs, errors, quality, quarantine, schema_changes, lineage, explain, metadata, streams, annotations, access, operations, cost): somente acréscimo, particionado por data do evento. Nunca UPDATE/DELETE. Cada execução/evento é uma nova linha imutável.
  • Estado (ctrl_ingestion_state): registrado como histórico somente de acréscimo; a marca d'água/estado atual por destino é a linha mais recente (row_number()/max last_updated_at_utc). Isto mantém o estado auditável e imutável como as tabelas de evidências.
  • Bloqueios (ctrl_ingestion_locks): melhor esforço, mapeado por adaptador; a única superfície inerentemente mutável (aquisição/liberação) e pode ser um mecanismo que não seja de tabela (gravação condicional do DynamoDB, etc.) por adaptador.
