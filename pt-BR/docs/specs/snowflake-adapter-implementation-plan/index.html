Plano de implementação do adaptador Snowflake

Propósito​

Este plano define como implementar contractforge-snowflake como um programa de primeira classe Adaptador ContractForge sem enfraquecer a arquitetura central.

É baseado nas lições do adaptador de referência Databricks e do AWS Adaptador Glue/Iceberg:

  • manter o núcleo livre de SDKs de plataforma e bibliotecas de tempo de execução;
  • fazer o adaptador consumir modelos e contratos principais públicos;
  • declare capabilities conservatively;
  • use execução nativa da plataforma, não execução com menor denominador comum;
  • mantenha uma biblioteca de tempo de execução estável como caminho de execução padrão;
  • usar artefatos gerados para revisão, implantação e depuração, não como o único fonte do comportamento de ingestão;
  • persista o esquema canônico de evidência/tabela de controle ContractForge e preencha-o com valores nativos do Snowflake.

O alvo de paridade Snowflake pesquisado é Capacidade Snowflake e paridade de evidências.

Regras de arquitetura não negociáveis​

  1. O núcleo não deve importar conector Snowflake, Snowpark, SQLAlchemy ou qualquer Snowflake SDK.
  2. O adaptador não deve alterar o significado do contrato para se adequar ao Snowflake.
  3. A configuração específica do Snowflake deve residir no environment.parameters.snowflake ou artefatos de implantação de propriedade do adaptador, não na semântica de ingestão.
  4. As importações em tempo de execução devem ser lentas. Importando contractforge_snowflake localmente para o planejamento não deve exigir credenciais Snowflake ou uma conexão de rede.
  5. A semântica não suportada ou sensível à revisão deve retornar REVIEW_REQUIRED ou UNSUPPORTED; nenhum substituto silencioso do modo histórico para anexar, da quarentena para avisar, ou de streaming para lote.
  6. As tabelas de evidências devem usar o esquema principal da tabela de controle. Conta Snowflake As visualizações de uso e esquema de informações são fontes de evidências, não substitutas para Evidência ContractForge.
  7. Use registros e objetos estratégicos para origem/gravação/evidência/governança roteamento. Evite if platform ==... longos ou if source_type ==... sequenciais dispatch blocks.
  8. Cada capacidade implementada precisa de documentos, testes, comportamento do planejador e comportamento de tempo de execução ou um item bloqueador/revisão explícito.

Isso reflete a divisão de domínio Databricks e AWS sem copiar seu tempo de execução tecnologia. O adaptador Snowflake deve começar menor que Databricks, mas ainda deve evitar arquivos divinos desde o primeiro dia.

Modelo de tempo de execução​

O modelo de execução Snowflake padrão deve seguir a correção AWS e o Databricks operational shape:

contract bundle
  -> core validation
  -> Snowflake planning
  -> publish Snowflake contract/environment/runtime artifacts
  -> create/update stable Snowflake runtime object
  -> runtime object loads contract and calls contractforge_snowflake runner
  -> runner executes source, preparation, quality, write and evidence steps

A ingestão por contrato gerada SQL não deve ser o padrão. Snowflake segue o modelo publicar e executar: implantar o contrato, o ambiente e o executor metadados e deixe um tempo de execução do adaptador estável executar a ingestão. Artefatos SQL são permitidos apenas para inicialização de infraestrutura e suporte de revisão. O adaptador deve expor um runner estável:

from contractforge_snowflake.runtime import run_ingestion

run_ingestion(
    contract_uri="stage://CONTRACTFORGE_ARTIFACTS/contracts/orders.json",
    environment_uri="stage://CONTRACTFORGE_ARTIFACTS/environments/prod.json",
)

A implantação pode implementar isso com um ou mais dos seguintes:

  • Snowflake Scripting stored procedure;
  • Procedimento armazenado Python com a roda adaptadora preparada;
  • comando de implantação externa que envia o executor SQL por meio do Snowflake conector;
  • gráfico de tarefas que chama o ponto de entrada do executor estável para cada etapa do contrato.

A primeira implementação pode construir pacotes de publicação e executá-los através de um servidor local. Python CLI que chama o mesmo tempo de execução do adaptador. O tempo de execução em Snowflake pode ser faseado, mas a arquitetura deve manter o modelo de executor de biblioteca como o caminho padrão, não geração de código por contrato.

Fase 3: Publicar pacote e manifesto de tempo de execução​

Scope​

Crie artefatos de publicação revisáveis ​​para:

  • instantâneo bruto do contrato;
  • environment snapshot;
  • runner invocation manifest;
  • evidência DDL do esquema principal da tabela de controle;
  • notas de revisão de estágio, armazém, função e tarefa;
  • manifesto de implantação.

Publicar regras​

  • Os nomes dos artefatos de publicação devem ser determinísticos.
  • Valores sensíveis devem ser redigidos.
  • Seções não suportadas produzem bloqueadores de revisão ou avisos, e não artefatos de ingestão parcial executáveis.
  • O pacote de publicação padrão não deve conter .source.sql, .write.sql ou outra lógica de ingestão por contrato.
  • O despacho em tempo de execução deve usar registros:
SOURCE_RUNNERS = {
    "table": TableSourceRunner(),
    "view": TableSourceRunner(),
    "sql": SqlSourceRunner(),
    "csv": StageFileSourceRunner(),
}

Acceptance​

  • Cada artefato possui um tipo, caminho, bytes, contagem de linhas e finalidade no manifesto.
  • Publish manifests declare execution_model: library_runner.
  • Os testes falharão se o SQL de origem/gravação gerado for emitido pelo caminho padrão.
  • Nenhuma cadeia if de origem/modo de gravação sequencial se transforma em um runner divino.

Fase 5: Modos de Gravação​

Modos de tempo de execução iniciais​

ModoImplementation
appendINSERT INTO target SELECT... ou COPY INTO target.
overwritepadrão de transação revisado: CREATE OR REPLACE TABLE AS SELECT ou truncar/inserir com base na política ambiental.
upsertMERGE INTO target USING staging.
hash_diff_upserta projeção de hash de teste mais MERGE apenas alterou as linhas.
snapshot_reconcile_soft_deleterenderizar revisão primeiro SQL; execute somente quando a integridade da fonte for declarada.
historicalrevisão necessária até que a política histórica seja explícita e testada.

Hard Requirements​

  • As chaves duplicadas devem falhar antes do MERGE.
  • As chaves de mesclagem nulas devem seguir a política principal.
  • A diferença de hash deve excluir automaticamente colunas geradas/estrutura.
  • O usuário pode declarar hash_keys para tabelas amplas.
  • O usuário pode declarar hash_exclude_columns para hashing de todas as colunas.
  • O adaptador não deve usar contadores de linha Snowflake sozinhos quando puder calcular valores exatos ContractForge counters in staging.

Tests​

  • Anexar linhas de inserções.
  • A substituição preserva o comportamento planejado e emite um aviso.
  • Upsert inserções e atualizações corretamente.
  • A diferença de hash não atualiza linhas inalteradas.
  • As atualizações de diferenças de hash alteraram as linhas.
  • revisão de retornos históricos necessária.

Fase 7: Qualidade e Quarentena​

Regras de tempo de execução​

Tipo de qualidadeImplementation

Enforcement​

  • fail: persiste evidência de qualidade/erro sempre que possível e falha na execução.
  • warn: persistir evidências de qualidade e continuar.
  • quarantine: apenas predicados em nível de linha removem linhas e gravam ctrl_ingestion_quarantine.
  • regras de quarentena agregadas tornam-se avisos registrados ou falham de acordo com o núcleo política; eles não devem inventar linhas em quarentena.

Integração nativa opcional​

As funções métricas de dados Snowflake podem enriquecer a observabilidade, mas não devem substituir aplicação da qualidade do tempo de execução até que o comportamento corresponda ao ContractForge falha/aviso/ quarantine semantics.

Query Tag​

Cada instrução de propriedade do adaptador deve definir uma tag de consulta estruturada:

{
  "product": "contractforge",
  "adapter": "snowflake",
  "run_id": "<run-id>",
  "project": "<project>",
  "target": "<database.schema.table>"
}

Esta é a chave de junção para reconciliação pós-execução.

Acceptance​

  • O sucesso grava evidências de execução/estado/metadados/linhagem.
  • A falha grava evidências de execução/erro e aumenta novamente ou retorna o status de falha.
  • A quarentena de qualidade grava evidências de qualidade e de quarentena.
  • A reconciliação de custos pode ser executada após a latência de uso da conta e anexar Linhas ctrl_ingestion_cost.

Fase 10: Implantação do Projeto​

Fluxo do Projeto​

contractforge-snowflake deploy-project project.yaml --target dev --dry-run
contractforge-snowflake deploy-project project.yaml --target dev --apply
contractforge-snowflake run-project project.yaml --target dev --wait

Responsabilidades de implantação​

  • Load project.yaml.
  • Resolva o ambiente e as conexões por meio do núcleo.
  • Planeje cada contrato.
  • Bloqueie contratos não suportados.
  • Renderizar artefatos.
  • Faça upload de artefatos para preparar ou configurar o artefato URI.
  • Criar/atualizar procedimento de tempo de execução ou scripts SQL.
  • Crie/atualize tarefas e dependências.
  • Optionally resume/suspend tasks.
  • Registre o manifesto de implantação.

Schedule Mapping​

O núcleo project.schedule permanece neutro em termos de plataforma:

schedule:
  cron: "0 6 * * *"
  timezone: America/Sao_Paulo

Snowflake maps it to task syntax:

SCHEDULE = 'USING CRON 0 6 * * * America/Sao_Paulo'

As dependências de tarefas são mapeadas para relacionamentos AFTER. O adaptador deve documentar Limites do gráfico de tarefas Snowflake e comportamento de falha na revisão da implantação.

Fase 11: Execução em Tempo de Execução​

Tempo de execução inicial​

  • local CLI opens Snowflake session;
  • creates temp/staging objects;
  • carrega artefatos de contrato/ambiente publicados;
  • chama o mesmo tempo de execução contractforge_snowflake usado pelas tarefas implementadas;
  • grava evidências usando o escritor de evidências do adaptador;
  • optionally reconciles query history by query tag.

Tempo de execução estável no Snowflake​

  • empacotar os metadados da roda do adaptador e do executor em um estágio;
  • create procedure/function runner;
  • gráfico de tarefa chama executor estável com contrato URI;
  • o executor carrega o contrato e o artefato do ambiente;
  • runner executa as mesmas etapas de tempo de execução do adaptador.

Acceptance​

  • Um contrato pode ser executado por CLI local.
  • Um projeto pode ser executado no gráfico de tarefas.
  • Atualizar apenas o artefato do contrato altera o comportamento da próxima execução sem regenerar ingestion logic.

  • ctrl_ingestion_cost after reconciliation;
  • query tag join to QUERY_HISTORY;
  • task graph join to TASK_HISTORY.

Cost Discipline​

Use o menor armazém e suspenda-o após os testes. Testes de alto custo devem ser explicitamente bloqueado como testes de smoke AWS. Nenhum teste deve criar uma tarefa de longa duração sem um comando de limpeza.

PlanejamentoCobertura completa de suporte, aviso, revisão e não suportado.
RenderizaçãoArtefatos e manifestos determinísticos.

Governançacomentários/tags/concessões validados; as políticas de linha/máscara revisam ou aplicam o caminho testado.

  1. ADR e extensions-snowflake.md.
  2. Esqueleto de pacote e testes de limite de importação.
  3. Registro de capacidade e testes de planejador.
  4. Módulos de identificador, nomenclatura e ambiente.
  5. Renderização para SQL/table/append/overwrite/upsert.
  6. Evidência DDL e escritor.
  7. Execução em tempo de execução local através do conector Snowflake.
  8. Verificações de qualidade e quarentena.
  9. Política de esquema.
  10. Anotações, comentários e tags.
  11. Implantação de projetos e renderização de gráficos de tarefas.
  12. Testes smoke reais no Snowflake.
  13. Reconciliação de custo/histórico de consultas.
  14. Subvenções/políticas de governação.
  15. Atualizações de site/docs/AI.
  16. Estabilização e candidato a liberação.

Work Breakdown​

Marco A: Adaptador somente para planejamento​

Escopo estimado: 2 a 4 dias.

  • esqueleto de pacote;
  • registro de capacidades;
  • plan_snowflake_contract;
  • documentos e testes.

Feito quando os contratos principais podem ser avaliados para Snowflake com determinística resultados de planejamento e nenhuma dependência Snowflake instalada.

Marco B: Adaptador de pacote de publicação​

Escopo estimado: 4 a 7 dias.

  • geração de bundle de publicação;
  • evidência de renderização DDL;
  • relatórios de revisão;
  • manifestos de implantação.

Feito quando os projetos de paridade Databricks/AWS podem construir pacotes de publicação Snowflake e itens necessários para revisão sem execução. Os artefatos padrão não devem conter ingestão por contrato SQL.

Marco C: Tempo de execução local​

Escopo estimado: 1 a 2 semanas.

  • dependência opcional do conector Snowflake;
  • tratamento de sessão/segurança;
  • append/overwrite/upsert;
  • redator de evidências;
  • falha/aviso/quarentena de qualidade;
  • esquema estrito/aditivo.

Feito quando testes de smoke reais do Snowflake podem ser executados a partir do CLI local.

Marco D: Implantação do Projeto​

Escopo estimado: 1 a 2 semanas.

  • artefatos de stage;
  • procedimento armazenado ou implantação de executor estável;
  • implantação de gráfico de tarefas;
  • suporte a agendamento, fuso horário e dependências;
  • comando executar/esperar/status.

Feito quando o fluxo no nível do projeto funciona sem que os usuários executem arquivos SQL individuais manualmente.

Marco E: Estabilização​

Escopo estimado: 1 a 2 semanas.

  • evidência do caminho da falha;
  • reconciliação de custos;
  • revisão/aplicação da governança;
  • ai/docs;
  • empacotamento de release.

Feito quando Snowflake tem a mesma disciplina de lançamento que AWS: testes reais, evidências auditoria e uma matriz de estabilização.

Principais riscos​

RiscoMitigação
A latência de uso da conta atrasa as linhas de evidência/custo.Escreva evidências ContractForge imediatas durante a execução; adicione comando de reconciliação para enriquecimento de uso da conta.
o histórico parece possível, mas a semântica é específica do projeto.Mantenha o REVIEW_REQUIRED até que as políticas de chegada tardia, exclusão e data de vigência sejam explícitas e testadas.
Snowflake pode executar SQL, mas não a extração geral.Trate JDBC/HTTP/Kafka como necessário para revisão, a menos que use padrões de conector pré-estágio/nativos.
O SQL gerado cresce no tempo de execução.Mantenha o alvo do runner estável; O SQL gerado permanece como artefato de revisão/depuração/implantação.
As políticas de governança variam de acordo com a hierarquia de papéis e privilégios.Iniciar somente validação/revisão primeiro; exigem comandos de aplicação explícitos.
Testes de custo consomem créditos.Use warehouse mínimo, controle de teste explícito e limpeza.
Snowpark incentiva dependências mais amplas.Mantenha Snowpark opcional e apenas para semântica que SQL não pode preservar.

Definição de pronto​

contractforge-snowflake está pronto para alfa quando:

  1. tem uma wheel independente;
  2. o caminho de importação padrão não depende do SDK Snowflake;
  3. o planejador retorna status corretos para todas as seções principais do contrato;
  4. anexar, substituir e executar upsert do estado atual em uma conta Snowflake real;
  5. falha/aviso/quarentena de qualidade funciona para tipos de regras suportados;
  6. tabelas de controle canônico são criadas e preenchidas;
  7. tags de consulta unem evidências de execução do ContractForge ao histórico de consultas do Snowflake;
  8. a implantação do projeto suporta cronograma, fuso horário e dependências;
  9. a semântica não suportada/exigida para revisão é explícita;
  10. documentos, site, IA e matriz de estabilização são atualizados.
