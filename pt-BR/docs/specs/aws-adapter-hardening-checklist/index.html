Lista de verificação de endurecimento do adaptador AWS

Propósito​

O adaptador AWS deve evoluir com o mesmo formato de engenharia do adaptador de referência Databricks:

  • limite estrito de núcleo/adaptador
  • importação de plataforma opcional SDK
  • capacidade explícita e limites de revisão
  • domain-split modules instead of god files
  • comportamento de segurança aplicado antes da renderização ou execução em tempo de execução

Esta lista de verificação registra o trabalho de proteção de curto prazo antes de adicionar amplo recurso de tempo de execução do AWS.

Guarda-corpos necessários​

AreaAção necessáriaAcceptance
Testes de arquiteturaAdicionar tests/test_aws_architecture.py, espelhando os testes de arquitetura do Databricks.Os testes provam que o núcleo não importa AWS, as importações de AWS SDK são preguiçosas/opcionais e os módulos AWS permanecem abaixo do limite de tamanho de arquivo acordado.

Interpretação da fonteAdicione contractforge_aws.sources.interpret para tradução de recursos de origem.source.intent: file_stream e source.type: incremental_files usam as mesmas regras explícitas de mapeamento AWS.

Near-Term Work Order​

  1. Add AWS architecture tests. Done.
  2. Divida a renderização de preparação AWS em módulos focados. Feito.
  3. Mova os auxiliares de tempo de execução Glue/S3 opcionais para trás de um namespace voltado para o tempo de execução, mantendo o API de alto nível pequeno. Feito.
  4. Adicione interpretação de origem AWS para file_stream e incremental_files. Feito.
  5. Adicione auxiliares de segurança HTTP/REST antes de renderizar trabalhos executáveis ​​HTTP/REST AWS. Concluído: http_file valida alvos e recusa segredos inline; rest_api lê o cliente REST principal (contractforge_core.connectors.read_rest_api_records/contractforge_core.connectors.api.rest), que valida alvos e ao qual o tempo de execução Databricks também delega.
  6. Promova grupos auxiliares de renderização/tempo de execução maduros em pastas de domínio. Concluído: anotações, governança, qualidade, preparação, esquema, estado, evidência e linhagem não são mais módulos simples do rendering/*_runtime.py.
  7. Harden gerou ordenação de evidências Glue e caminhos de falha. Concluído: a evidência de sucesso é gravada somente após o job.commit(), as falhas gravam a evidência de erro editada mais a evidência de execução com falha, e as falhas na gravação da evidência não mascaram a falha original do Glue.
  8. Extend parity docs after each new AWS semantic mapping.

Política de extensão de adaptador​

O comportamento da plataforma que o núcleo API não pode padronizar segue a ordem em adapter-parameter-policy.md: parâmetro semântico central → padrão seguro + aviso → extensão do adaptador → REVIEW_REQUIRED → UNSUPPORTED.

Prefira parâmetros portáteis a extensões de adaptador. Quando um recurso precisa de um contagem de parâmetros diferente por plataforma (por exemplo, uma plataforma precisa de uma, AWS duas), modelar o superconjunto de parâmetros portáteis no contrato principal com os somente AWS opcional (camada "Detalhe opcional principal"), em vez de buscar uma extensão. Teste de decisão: uma segunda plataforma poderia implementar este parâmetro sob o mesmo nome semântico? Sim → parâmetro principal opcional portátil (nomeado pela intenção, não pelo fornecedor API). Não → extensão do adaptador como último recurso.

As extensões do adaptador residem em environment.parameters.aws (ajuste/implantação) e extensions.aws (execução). Limites de segurança para qualquer extensão:

  1. Lista de permissões por adaptador; chaves desconhecidas avisam e nunca são respeitadas silenciosamente.
  2. Os valores de extensão passam pelo mesmo caminho de segredo/edição; nunca asse um credencial em um script Glue renderizado ou artefato publicado por S3.
  3. Os dados do contrato/extensão nunca devem relaxar um controle de segurança - apenas o operador sinalizadores de ambiente (por exemplo, CONTRACTFORGE_ALLOW_PRIVATE_HTTP_TARGETS) podem diminuir a postura. Um contrato não pode desabilitar a proteção SSRF ou forçar segredos embutidos.
  4. Caminhos/URLs de extensão obtêm a mesma validação de travessia/SSRF que os campos principais.
  5. Cada adaptador ignora os blocos de extensão de outros adaptadores.

Concluído: AWS lê extensions.aws, avisa sobre chaves desconhecidas com AWS_UNKNOWN_EXTENSION e documenta as chaves permitidas em extensions-aws.md.

Pronto: o caminho de autenticação rds_iam JDBC renderiza um tempo de execução _cf_rds_iam_token(...) chamada (boto3 rds.generate_db_auth_token) para o {{rds_iam_token}} do núcleo espaço reservado e remove os metadados contractforge.rdsIam*, para que o token seja gerado quando a tarefa é executada e nenhuma credencial é inserida no artefato.

Non-Goals​

  • Não mova as importações AWS SDK para o núcleo.
  • Não faça o adaptador AWS imitar os detalhes de implementação do Databricks quando a semântica do AWS for diferente.
  • Não marque um recurso suportado apenas porque Glue, Athena, Lake Formation ou Iceberg possuem uma primitiva relacionada.
  • Não continue adicionando comportamento a um módulo grande quando um arquivo renderizador/tempo de execução específico do domínio estiver mais claro.
