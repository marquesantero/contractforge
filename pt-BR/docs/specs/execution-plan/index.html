Especificação do Plano de Execução

Propósito​

Um plano de execução é um plano abstrato produzido pelo núcleo semântico. Não é um trabalho Databricks, um script Glue, um pipeline Fabric ou um arquivo SQL.

O plano existe para que os adaptadores possam renderizar artefatos nativos enquanto preservam a semântica do contrato declarado.

Conteúdo do plano​

Um plano de execução pode conter:

  • etapa de leitura da fonte
  • etapa de forma, quando a intenção de forma é declarada
  • etapa de transformação, quando a intenção de transformação é declarada
  • etapa de validação de qualidade
  • passo de gravação
  • etapa de aplicação de governança
  • etapa de registro de evidências

Rendering​

Os adaptadores renderizam planos de execução em artefatos nativos da plataforma, por exemplo:

  • YAML
  • SQL
  • Python
  • Databricks Asset Bundles
  • Glue job scripts
  • Fabric pipeline JSON
  • Terraform snippets
  • Markdown review reports

A execução é opcional e de propriedade do adaptador. O núcleo não executa código de tempo de execução da plataforma.

Resultados de Execução​

O núcleo possui um pequeno vocabulário de resultados de execução para tempos de execução do adaptador:

  • SUCCESS
  • FAILED
  • SKIPPED

Um adaptador pode incluir métricas nativas e a instrução ou artefato executado, mas o status do resultado e os campos básicos permanecem portáteis para que evidências, novas tentativas e painéis possam usar o mesmo vocabulário em todas as plataformas.

Prepared Inputs​

O núcleo define um pequeno modelo de entrada preparada para tempos de execução do adaptador:

  • visualização de origem ou nome da relação preparada
  • colunas de origem
  • linhas lidas
  • linhas em quarentena
  • nome da fonte
  • metadados de origem

Os adaptadores podem preparar esta entrada com Spark, SQL, pipelines nativos ou etapas de aterrissagem externas. O núcleo define apenas o formato de transferência e regras comuns de fallback, como derivar linhas escritas a partir de métricas de execução ou contagens de linhas preparadas.

Janelas de execução​

O núcleo possui planejamento de janela de execução para execuções particionadas ou de preenchimento:

  • geração de janela de tempo desde início/fim/cada
  • child run idempotency key composition
  • parâmetros de tempo de execução filho
  • resumos de resultados de janelas agregadas

Os adaptadores renderizam filtros específicos da plataforma e artefatos de tempo de execução para cada janela. Por exemplo, Databricks renderiza predicados de carimbo de data/hora SQL com aspas do identificador Databricks, enquanto outros adaptadores podem renderizar Snowflake SQL, expressões Spark, filtros Dataflow ou parâmetros de trabalho nativos.

Parity Catalogs​

O núcleo define modelos de catálogo de paridade de plataforma neutra para comparar mecanismos de plataforma nativa com a semântica ContractForge:

  • id e título do cenário
  • modo de gravação solicitado
  • mecanismo nativo candidato
  • expectativa: must_match, intentional_difference ou unsupported
  • capacidades necessárias e campos de contrato
  • semântica esperada
  • metric expectations
  • bloqueadores ou notas de revisão

Os adaptadores possuem o catálogo de cenários real para seus mecanismos nativos. Databricks, por exemplo, pode publicar cenários de paridade para SQL MERGE e Lakeflow AUTO CDC sem tornar esses motores parte do núcleo.

Escreva registros de estratégia​

O núcleo define um pequeno registro de estratégia de gravação para decisões do adaptador:

  • strategy kind
  • selected engine
  • reason
  • blockers
  • warnings

Os adaptadores possuem a lógica de decisão e os nomes dos mecanismos. Por exemplo, Databricks pode escolher Delta anexado, SQL MERGE, Lakeflow AUTO CDC ou um algoritmo ContractForge Delta; outro adaptador pode escolher um mecanismo nativo diferente, preservando o mesmo formato de registro de estratégia.

Regras de segurança​

O planejador não deve:

  • fazer downgrade silenciosamente dos modos de gravação
  • apagar requisitos de governança
  • substitua histórico por anexar
  • assume streaming equivalence across engines
  • gerar código específico de tempo de execução do pacote principal
